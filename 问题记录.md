## 详细问题

### 问题 1: DomainIndex.initialize() RuntimeWarning

**严重度**: HIGH（启动报错，影响体验）
**发现**: 每次启动都报 `RuntimeWarning: coroutine 'DomainIndex.initialize' was never awaited`
**原因**: `cli.py:242` 在 `asyncio.run(main())` 内部调用 `_aio.get_event_loop().run_until_complete(domain_index.initialize())`，但事件循环已在运行，不能再嵌套 run_until_complete
**位置**: `app/cli.py:242`
**修复方案**: 改成直接 `await domain_index.initialize()`

### 问题 2: one-shot 模式输出中文乱码

**严重度**: HIGH（one-shot 模式几乎不可用）
**发现**: one-shot 模式下终端输出全是乱码 `[ģħͺ] deepseek-chat`
**原因**: Windows 终端默认 GBK 编码，`_safe_print` 的容错处理不够
**位置**: `app/cli.py:37-43`
**修复方案**: 强制 sys.stdout 切换到 UTF-8，或在 one-shot 输出前设置编码

### 问题 3: system_prompt 缺少工作目录信息

**严重度**: HIGH（模型会把文件写到错误位置）
**发现**: 模型不知道当前工作目录，文件写到了 `/root/` 或随机路径
**原因**: cli.py 的 system_prompt 只说了模型名和能力，没有注入 `os.getcwd()`
**位置**: `app/cli.py:197-207`（prompt_builder.set_system_prompt）和 `app/cli.py:301-311`（context.system_prompt）
**修复方案**: 在 system_prompt 中注入 `当前工作目录: {os.getcwd()}`

### 问题 4: bash 工具超时问题

**严重度**: MEDIUM（交互式程序无法测试）
**发现**: TODO 应用因为用 input() 等待输入，bash 超时退出码非 0
**原因**: bash 工具默认 120s 超时，但交互式程序永远等不到输入
**位置**: `app/tools/bash.py`
**修复方案**: 不需要修 bash 本身，但 system_prompt 应该告诉模型不要运行交互式程序，或者用管道输入

## 修复记录

### 修复 1: DomainIndex.initialize() RuntimeWarning

**改了什么**: `app/cli.py:242`
**改动**: 把 `_aio.get_event_loop().run_until_complete(domain_index.initialize())` 改成直接 `await domain_index.initialize()`
**原因**: 在 `asyncio.run(main())` 内部不能再嵌套 `run_until_complete()`，应该直接 await
**验证**: 启动不再报 RuntimeWarning，pytest warnings 从 10 个降到 8 个

### 修复 2: one-shot 模式输出中文乱码

**改了什么**: `app/cli.py:main()` 函数开头
**改动**: 添加 `sys.stdout.reconfigure(encoding="utf-8", errors="replace")` 和 `sys.stderr.reconfigure(...)`
**原因**: Windows 终端默认 GBK 编码，中文、emoji 都打不出来。reconfigure 强制 UTF-8
**验证**: one-shot 模式输出中文正常显示，包括 `[模型]` `[多Agent]` 等标签

### 修复 3: system_prompt 缺少工作目录信息

**改了什么**: `app/cli.py` 的两个 system_prompt 构建处
**改动**: 在 system_prompt 末尾追加：
- `当前工作目录: {os.getcwd()}`
- `操作系统: Windows`
- 使用绝对路径或基于工作目录的相对路径
- 不要运行需要交互式输入的程序

同时把 `context.metadata["project_path"]` 从 `"."` 改为 `os.getcwd()` 的绝对路径

**原因**: 模型不知道自己在哪，会把文件写到 `/root/` 之类的错误路径
**验证**: 模型现在正确写到 `C:\Users\12433\Desktop\myagent\controllable-agent\` 目录

### 修复 4: bash 运行交互式程序会卡住

**改了什么**: 同修复 3，在 system_prompt 中追加提示
**改动**: 告诉模型"不要运行需要用户交互式输入的程序（如 input()），改为接受命令行参数或用管道输入"
**原因**: bash 工具无法提供交互式输入，程序会卡住直到超时
**验证**: 模型创建计算器时用了命令行参数而非 input()，4 个测试全部正确运行

## 第二轮测试（T11-T15）

测试日期: 2026-05-04（续）

### 问题 5: 多 Agent 子循环 max_tool_calls_per_turn 太小

**严重度**: HIGH（多 Agent 协作模式几乎不可用）
**发现**: 5 个子 Agent 中，除了 planner（写 plan.md 只需少量工具调用），其他 Agent 全部触发 "Max tool calls (10) reached, forcing exit"
**原因**: `runtime.py:404` 创建子 Agent 循环时只设了 `max_turns`，没设 `max_tool_calls_per_turn`，用了默认值 10。Coder 任务（写多个文件）通常需要 20+ 次工具调用
**位置**: `agent/runtime.py:404`
**修复方案**: 在 `ROLE_MAX_TURNS` 中同时配置各角色的 `max_tool_calls_per_turn`，或在 spawn 时根据角色类型设更大的值

### 问题 6: 非 TUI 环境输出 emoji 导致 GBK 崩溃

**严重度**: MEDIUM（影响程序化调用多 Agent 编排）
**发现**: 直接用 Python API 调用 `runtime.orchestrate()` 时，子 Agent 输出包含 emoji（✅），打印到 stdout 触发 `UnicodeEncodeError: 'gbk' codec can't encode character '✅'`
**原因**: 非 TUI 环境没有执行 `sys.stdout.reconfigure(encoding="utf-8")`，Windows 默认 GBK
**位置**: `agent/runtime.py` 的 SubAgentResult.output 输出
**修复方案**: 方案 1：在 runtime.orchestrate() 内部对所有输出做 sanitize；方案 2：在 spawn() 中确保子 Agent 的 context 也注入 UTF-8 设置；方案 3：使用 errors="replace" 处理输出

## 第二轮修复记录

### 修复 5: 多 Agent 子循环 max_tool_calls_per_turn 太小

**改了什么**: `agent/role_prompts.py` 和 `agent/runtime.py`
**改动**:
1. 在 `role_prompts.py` 新增 `ROLE_MAX_TOOL_CALLS` 字典，为每个角色配置独立的 `max_tool_calls_per_turn`
2. 在 `runtime.py` 新增 `_get_max_tool_calls()` 静态方法
3. 在 `spawn()` 创建子 Agent 循环时，同时设置 `max_tool_calls_per_turn`

具体值：
- coordinator: 15（主要是读+写 decisions.md）
- planner: 20（需要读文件+写 plan.md）
- coder: 30（需要大量 write/edit/bash 操作）
- reviewer: 20（需要 read+glob+grep 验证）
- memorizer: 10（主要是读+写总结）

**原因**: 子 Agent 默认 `max_tool_calls_per_turn=10`，Coder 经常需要 20+ 次工具调用来完成复杂任务，10 次就强制退出了
**验证**: pytest 449 tests passed

### 修复 6: 非 TUI 环境输出 emoji 导致 GBK 崩溃

**改了什么**: `agent/runtime.py` 的 `spawn()` 方法
**改动**: 在构建 `SubAgentResult` 前，对 `result.final_output` 做 UTF-8 安全编码处理：
```python
raw_output = result.final_output if result else ""
safe_output = raw_output.encode("utf-8", errors="replace").decode("utf-8", errors="replace")
```

**原因**: Windows 非 TUI 环境 stdout 是 GBK 编码，子 Agent 输出含 emoji（✅ ❌）时打印崩溃
**验证**: pytest 449 tests passed

## 第三轮测试（M1-M3 多 Agent 编程测试）

测试日期: 2026-05-04（续续）

### 问题 7: Reviewer 没有实际运行测试验证

**严重度**: HIGH（Reviewer 角色形同虚设，不能保证代码质量）
**发现**: M1 中 reviewer 报告 completed 但代码有类型注解 bug（`list[Task]` 遮蔽内置类型）和 fixture bug（Path vs str），都没有被发现。M3 中 reviewer 也报 completed 但有 2 个测试失败
**原因**:
1. reviewer.yaml 的 system_prompt 太笼统（"模块级快速验证"），没有明确要求运行 pytest
2. orchestrate() 中 Phase B/C 的任务描述太简短，没有具体操作步骤
3. reviewer 说"不能自己改代码"，和"执行集成测试"矛盾——测试需要用 bash 跑

**修复**:
1. 更新 `.agent-base/agents/reviewer.yaml` — 添加具体检查清单（类型注解、方法名遮蔽、运行 pytest）
2. 更新 `agent/runtime.py` orchestrate() — Phase B 的 reviewer 任务描述加入"运行已有测试验证通过"；Phase C 加入具体步骤（glob 找目录、运行 pytest、分析失败原因）
3. 提升 reviewer 的 max_tool_calls 从 20 到 25

**验证**: M3 测试中 reviewer Phase C 用了 25 次工具调用（之前只有 16 次），说明确实做了更多验证工作

### 问题 8: 多 Agent 代码质量缺陷（系统性问题）

**严重度**: MEDIUM（不是框架 bug，但影响多 Agent 输出质量）
**发现**:
- M1: `list[Task]` 类型注解——方法名 `list()` 遮蔽了内置 `list` 类型
- M1: fixture 中 `temp_path` 变量类型不一致（str vs Path）
- M3: `str(error)` 不包含 status_code（message 覆盖了默认值）
- M3: retry jitter 可能超过 max_delay（没有 clamp）

**分析**: 这是 LLM 代码质量问题，不是框架问题。但在 5 Agent 协作流程中，Review 层应该能捕获这些。根本原因是 reviewer 和 coder 并行运行（Phase B），reviewer 在 coder 还没写完代码时就开始了，看不到最终代码

**改进方向**:
- 让 Phase B 的 reviewer 在 coder 完成后再启动（串行而非并行）
- 或者在 Phase C 的集成测试中更严格要求 reviewer 运行 pytest 并检查失败原因
- 已通过改进 reviewer prompt 和 orchestrate 任务描述做了部分改善

**改进方向**: 在 orchestrate() 完成后，由主循环做一次统一的记忆提取；或者在 Phase D 的 memorizer 任务中明确要求它写入 digest 文件

### 问题 11: 记忆提取时机过早

**严重度**: LOW
**发现**: d_001.md 的摘要说"测试正在运行中，尚未显示最终通过/失败结果"，说明记忆提取发生在最终结果出来之前
**原因**: AgentLoop.run() 在循环结束后立即提取记忆，但测试可能还在异步运行
**改进方向**: 延迟提取或等待所有工具调用完成后再提取
