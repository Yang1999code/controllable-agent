## 三、问题与分析

### 问题1: YAML 被 register_agent_type 覆写

**现象**: 测试中调用了 `register_agent_type(AgentTypeConfig(...))`，该方法会调用 `config.to_yaml()` 将简化配置写回 YAML 文件，覆盖了我们精心编写的角色配置。

**根因**: `register_agent_type` 的设计初衷是"注册并持久化"，但 Phase 3 中 YAML 是权威配置源，不应被运行时注册覆盖。

**修复**: `register_agent_type` 改为仅在 YAML 文件不存在时才写入。即 YAML 文件是权威的，运行时注册只是加载到内存。

### 问题2: 测试中变量解构顺序错误

**现象**: 集成测试中 `_make_system()` 返回 `(runtime, factory, shared_space)`，但部分测试用了 `factory, _, _` 的解构顺序，导致 `factory` 实际上是 `runtime` 对象。

**根因**: 解构时三个元素都是 `object` 类型，Python 不做类型检查。

**修复**: 调整解构顺序，使用 `_, factory, _` 取第二个元素。

### 问题3: MockProvider 无法模拟 spawn 失败

**现象**: 想测试"Planner 失败时 orchestrate 应立即返回"，但 MockProvider 总是返回文本内容，spawn 总是成功。

**分析**: MockProvider 的响应被当作正常文本，LLM 不调用工具也能"完成"任务。这是测试框架的局限性。

### 问题4（代码审查发现）: cross_agent_read 安全漏洞

**现象**: CrossAgentReadTool 缺少 agent_id 格式校验和 `..` 路径穿越防护。恶意输入如 `agent_id="../etc"` 或 `path="agent_view/../../etc/passwd"` 可能绕过白名单。

**修复**: 添加 `_AGENT_ID_PATTERN` 正则校验（只允许字母数字下划线连字符）+ `_PATH_TRAVERSAL_PATTERN` 检测 `..`。新增 3 个安全测试。

### 问题5（代码审查发现）: DelegateTaskTool 未传递 current_depth

**现象**: `delegate_task.py` 调用 `runtime.spawn()` 时没有传递 `current_depth`，默认为 0。导致 Coordinator 生成的子 Agent 从 depth=0 开始，嵌套深度控制完全失效。

**根因**: Phase 3 新增了 `depth` 到 context.metadata，但 DelegateTaskTool 没有从中提取。

**修复**: 添加 `current_depth=context.metadata.get("depth", 0)` 参数。

### 问题6（代码审查发现）: CLI 访问私有属性 + shared_space 未初始化

**现象**: `cli.py` 中 `tools._tools` 访问了私有属性；`shared_space.initialize()` 从未被调用。

**修复**: 改为 `dict(tools.tools)` 使用公共属性；在 CLI 中添加 `await shared_space.initialize()`。

### 问题7（代码审查发现）: initialize() 污染 issues.md + 双信号量

**现象**: `shared_space.initialize()` 向 issues.md 写入 `[INIT]` 条目，污染 Reviewer 的问题追踪。`spawn_parallel()` 创建了独立于 runtime `_semaphore` 的局部信号量，实际并发度被两层限制。

**修复**: initialize() 只写 `# 问题记录\n` 标题；spawn_parallel() 移除局部信号量，直接使用 runtime 的 `_semaphore`。

**处理**: 调整测试为"只注册 planner 不注册其他角色时，Phase B 的 agent_type 找不到会 rejected"。
