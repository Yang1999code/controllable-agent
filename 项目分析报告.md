# Controllable Agent 项目深度分析

## 一句话定位

> **"一个人就能管一支 AI 开发团队"** — 5 个专业 Agent 角色（Coordinator/Planner/Coder/Reviewer/Memorizer）分工协作，Wiki 式记忆自进化，纯 Python 手写、零框架依赖的多 Agent 协作系统。

GitHub: `Yang1999code/controllable-agent` | 作者: 国内个人开发者 | 协议: MIT | 代码量: ~8400 行（30 个模块）

## 1. 整体架构


## 2. 核心模块逐一解读

### 2.1 AgentLoop（loop.py, 475 行）— 主循环

**双层循环结构**（参考 Anthropic Pi Agent 的实现）：

```
外层 followUp 循环         内层 tool_calls 循环
┌─────────────────┐       ┌──────────────────┐
│ while turn < max│       │ while tc < max   │
│  get next msg   │──►    │  LLM.stream()    │
│  build prompt   │       │  parse tool_call │
│  compaction     │       │  execute tools   │
│  [内层循环]     │◄──    │  inject results  │
│  checkpoint     │       │  check overflow  │
│  nudge check    │       │  compact if need │
│  memory extract │       └──────────────────┘
└─────────────────┘
```

**关键数据流**：
```python
async def run(self, user_input: str, context: Context) -> AgentResult:
    # 1. 加载 CLAUDE.md（项目上下文）
    # 2. 外层 followUp 循环
    #    a. 检查 steer 消息（优先级高于 followUp）
    #    b. 动态构建 system prompt（Phase 2）
    #    c. 上下文溢出检测 → 3 层压缩
    #    d. KV Cache 断点检测
    #    e. 内层 tool_calls 循环
    #       - LLM 流式调用
    #       - 工具执行（并发）
    #       - 子 Agent 收件箱检查
    #    f. turn_end Hook + 检查点 + Nudge
    #    g. 记忆提取（异步，不阻塞主循环返回）
```

**与其他 Agent 循环的对比**：
- vs Claude Code: CCB 循环更复杂（1776行），本项目的双层结构更清晰易懂
- vs Hermes: Hermes 循环 12000+ 行，本项目 475 行就实现核心逻辑
- vs Pi Agent: 参考了 Pi Agent 的 followUp + tool_calls 双层设计

### 2.2 AgentRuntime（runtime.py, 828 行）— 多 Agent 运行时

这是整个项目最具创新性的模块。实现了一个完整的**多 Agent 操作系统**。

**Agent 类型注册系统**：
```python
# 5 种预设角色（通过 YAML 定义）
MULTI_AGENT_ROLES = ("coordinator", "planner", "coder", "reviewer", "memorizer")

# 每种角色有独立的：
# - system_prompt（角色人设）
# - tools_whitelist/blacklist（可用工具）
# - max_turns（最大轮次）
# - max_tool_calls_per_turn（每轮工具调用上限）
```

**并发控制**：
```python
self._semaphore = asyncio.Semaphore(max_concurrent)  # 默认 3
# 子 Agent 不能调用 delegate_task（防止套娃）
# 嵌套深度硬限制：max_depth=2
```

**Agent 间通信**：
```python
# 每个 Agent 有独立 inbox（asyncio.Queue）
# send_message(from, to, content) → check_inbox(agent_id)
# 主循环在每次工具执行后检查收件箱
```

**自动 Agent 选择**：
```python
def select_agent(self, task: str) -> str | None:
    # 对任务描述和 Agent 描述做 jieba 分词
```

### 2.3 orchestrate() — 分阶段编排（最核心的方法）

这是整个项目最精彩的部分，实现了**软件工程全流程的 Agent 自动化**：

```
Phase A:  Planner 串行 → 分析需求，写 plan.md
Phase A+: 分析 plan.md 复杂度，动态调整 max_tool_calls
Phase B-1: Coordinator + Memorizer 并行（轻量辅助）
Phase B-2: Coder 串行 → 按 plan.md 实现代码
Phase B-3: Reviewer 串行 → 审查代码，写 issues.md
Phase C:   Reviewer 总体集成测试（串行）
           ↓ 不通过？回到 Phase B（最多 3 轮）
Phase C+:  Reviewer 对抗测试（红队审查，故意找 bug）
Phase Fix+Verify: Coder 修复 → Reviewer 验证（最多 2 轮）
Phase D:   Memorizer 总结 + 技能结晶
Phase Memory: 统一记忆提取
Phase Skill: 技能结晶持久化
```

**打回机制**：
- Coder+Reviewer 配对内：模块级打回不限次数（内部解决）
- 总体集成测试：最多 3 轮，超了就升级给用户
- 用户随时插话：Coordinator 接收，按紧急度分级处理

### 2.4 记忆系统 — Wiki 式双层结构

两套记忆系统并行：

**旧系统（L0-L4 层级）**：
```
.agent-memory/
├── index.md              ← L0 全局索引
└── {project}/
    ├── l1_navigation/    ← 场景导航
    ├── l2_facts/         ← 环境事实
    ├── l3_experience/    ← 任务经验
    └── l4_sessions/      ← 会话归档
```

**新系统（digest/wiki + 四域）**：
```
.agent-memory/
├── digest/d_001.md       ← 任务摘要（YAML frontmatter + Markdown）
├── wiki/python_stack.md  ← 知识页（同主题 ≥5 digest 合并）
├── domains/
│   ├── conversation/     ← 对话记录
│   ├── profile/          ← 用户档案
│   ├── agent_view/       ← Agent 视角
│   └── task/             ← 任务追踪
└── index.md              ← 全局关键词倒排索引
```

**核心亮点**：
```
任务完成 → LLM 提取 digest → 同主题 ≥5 → LLM 合并为 wiki
```
- **批量而非逐轮**：一个任务单元完成后才提取，不是每句话都存
- **专用小模型**：记忆提取用便宜模型，不占主 Agent 上下文
- **抗幻觉**：检索优先级 wiki → digest → 空 = 没说过 = 不编造

### 2.5 技能结晶系统（autonomous_memory.py + crystallizer.py）

```
任务成功 → should_crystallize() 启发式判断
  → 工具调用 ≥ 3 次 + 耗时 ≥ 5s + 未重复结晶
  → quality_score() 三维评分（清晰度/完整性/可操作性）
  → quality ≥ 60 → 持久化为 YAML 技能卡
  → 下次匹配到自动加载
```

**技能卡格式**：
```yaml
name: python_type_check_fix
description: 修复 Python 类型注解问题
trigger_condition: 类型检查 类型注解 方法名遮蔽
steps:
  - tool: grep
    args: {pattern: "def (list|dict|set|tuple|input)\("}
    description: 搜索被遮蔽的内置类型方法名
  - tool: edit
    args: {...}
    description: 替换为正确的类型注解
config:
  category: code_quality
  priority: 80
  tags: [python, typing, bugfix]
quality_score: 85.0
```

**质量衰减**：
```python
# 每次使用后更新分数：
# - 成功 +10，失败 -20
# - 用户反馈加权
# - 使用次数越多，分数变化越小
# - 低于 30 分且使用 ≥3 次 → 自动淘汰
```

### 2.6 上下文管理

**三层压缩（渐进式）**：
```
Layer 1 Prune:  清除 3 轮前的工具输出 → 替换为占位符（不调 LLM）
Layer 2 Summary: 压缩旧消息为结构化摘要（模板化）
Layer 3 Emergency: 从最早消息逐条删除（直到降到 98% 以下）
```

**KV Cache 断点检测**：
追踪 system prompt + tools + 消息列表的哈希前缀，当发现前缀变化时告知调用方缓存失效。

## 4. 代码亮点（值得学习的设计）

### 4.1 接口先于实现
每个模块都先定义 ABC/Protocol：
```python
class IAgentRuntime(ABC):
    @abstractmethod
    async def spawn(self, ...) -> SubAgentResult: ...
    @abstractmethod
    async def spawn_parallel(self, ...) -> list[SubAgentResult]: ...
```
实际实现类可以随时替换，测试时也方便 mock。

### 4.2 渐进式 Phase 架构
```python
class AgentLoop:
    def __init__(self,
        # Phase 1 必需
        provider, tools, hooks, config,
        # Phase 2 可选增强
        prompt_builder=None, inspector=None, capability_registry=None,
        # Phase 3 可选增强
        runtime=None, autonomous_memory=None, ...
    ):
```
新功能通过可选参数注入，不影响已有代码路径。

### 4.3 Hook 事件系统
```python
15 种事件类型: LOOP_START, TURN_START, LLM_CALL, STREAM_THINKING,
              STREAM_TEXT, TOOL_PROGRESS, CONTEXT_COMPACTION,
              TURN_END, SUBAGENT_START, SUBAGENT_STOP,
              TASK_COMPLETE, LOOP_END
```
Hook 链支持优先级排序 + 异常隔离（一个 Hook 崩了不影响其他的）。

### 4.4 全局入口 my_agent.py
```python
# 用户只需 import my_agent
from my_agent import AgentLoop, ToolRegistry, AgentRuntime, MemoryExtractor
```
类似 OpenAI Agents SDK 的 `from agents import Agent, Runner`，封装内部模块路径。
