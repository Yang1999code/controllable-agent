# 多智能体系统设计（Phase 3）

## 一、设计目标

在现有 AgentLoop 双层循环基础上，构建**多 Agent 并行协作系统**。三个核心需求：

1. **共享记忆** — 每个 Agent 有私有索引，通过 MD 文件做共识交流区，只读互看视角
2. **并行循环** — 外层 followUp 循环中同时启动多个子 Agent 循环（类似线程池）
3. **5 角色分工** — Coordinator / Planner / Coder / Reviewer / Memorizer，任务多时可复制

---

| `FactStore` | digest/wiki CRUD | 100% — 每个 Agent 独立实例 |
| `DomainIndex` | 四域管理 + 倒排索引 | 100% — 每个 Agent 独立实例 |

| `MemoryExtractor` | LLM digest 提取 | 100% — Memorizer 独占使用 |
| `Compaction` | 三层压缩 | 100% — Memorizer 角色用 |
| `AutonomousMemory` | 技能结晶 + Nudge | 100% — 统一路径到 agents/{id}/ 下 |

加载规则（参考 Claude Code CLAUDE.md）：
- **AGENT.md**：所有 Agent 启动时加载，提供项目级上下文
- **agent_context.md**：只有对应的 Agent 加载自己的（Coder 只看 coder_001/agent_context.md）
- **shared/skills/**：AGENT.md 中引用到的才加载，按需读取
- **wiki/digest**：更深层引用，Agent 需要时通过检索获取

## 四、5 角色 Agent 定义

### 4.1 Coordinator（协调者）— 全局管控 + 思维链

**定位**：整个多 Agent 系统的**管理者 + 外部思维链**。类比操作系统的进程管理器：它不干活，但它决定谁来干、怎么干、干得对不对。它是唯一有 Agent 创建/消除权限的角色，负责管理 Agent 间的合作流程、权限分配和嵌套深度控制。

```yaml
name: coordinator
description: 多Agent系统管理者——Agent创建/消除、权限管理、流程编排、嵌套深度控制
category: coordination
system_prompt: |
  你是 Coordinator Agent，整个多 Agent 系统的管理者和思维链。

  你不做任何具体任务（不写代码、不审查、不提取记忆）。你的职责是：

  1. **Agent 生命周期管理**（你的独有权限）：
     - 你是唯一有 delegate_task 权限的子 Agent
     - 根据任务需要，创建（spawn）合适的 Agent
     - 发现某个 Agent 卡住或失败 → 终止它
     - 发现任务量突增 → 创建更多同类 Agent（如 3 个 Coder 并行）
     - 发现 Agent 角色分配不当 → 终止后重新分配
     - **严格限制嵌套深度 ≤ 2 层**：你创建的 Agent 不能再创建子 Agent
     - 统计当前活跃 Agent 数量，防止资源耗尽

  2. **权限管理**：
     - 每个角色的工具权限在配置中预定义，你负责检查是否越权
     - Coder 在写 Planner 该写的计划 → 角色越界，写入 decisions.md 警告
     - Reviewer 尝试改代码 → 权限违规，写入 decisions.md 警告
     - 定期检查各 Agent 的工具使用记录是否合规

  3. **流程监控与思维链**：
     - 持续监控当前协作流程是否正确推进
     - 检查各 Agent 状态是否卡死/超时/重复
     - 检查 shared/plan.md 的执行进度
     - 识别死锁（两个 Agent 互相等待）
     - 确保信息流方向正确：Planner → Coder → Reviewer → Memorizer

  4. **全局数据循环控制**：
     - 检查 shared/ 目录下文件是否一致（plan vs status vs issues）
     - 检测循环：配对内模块级打回不限次数；总体集成测试超过 3 轮 → 标记 ESCALATE
     - 防止数据无限膨胀（status 文件过大 → 提醒压缩）

  5. **用户中途补充信息处理**（核心能力）：
     - 每轮检查 shared/interrupts/ 目录，看用户有没有补充信息或新要求
     - 读取补充信息后，评估对当前正在运行的任务的影响：
       - 不影响方向 → 记录到 decisions.md，Agent 继续原计划
       - 需要调整 → 写入 decisions.md，通知 Planner 调整计划
       - 紧急打断 → 通过 inbox 通知相关 Agent，必要时终止当前 Agent 重新开始
     - 补充信息的处理优先级高于常规监控
     - 处理完后将 interrupt 文件标记为已处理（重命名为 intr_xxx.done.md）

  输出方式：
  - 常规观察 → 写入 shared/decisions.md
  - Agent 管理操作 → 直接使用 delegate_task 工具创建/终止 Agent
  - 紧急升级 → 通知主 Agent（通过 inbox），主 Agent 决定是否人工介入
  - **你不运行代码（bash），不编辑文件内容（只读写 shared/ 和 status）**

  关键约束：
  - 你创建的 Agent（coder/reviewer 等）不能再创建子 Agent（max_depth=2）
  - 你自己也是子 Agent，不能创建另一个 Coordinator（防止管理套娃）
  - 并发由 asyncio.Semaphore 控制（max_concurrent 配置），你不需要自己数数
  - 任务拆得够细时，可以创建多个 Coder/Reviewer 并行跑，小模块级别的快速迭代

tools_whitelist: [read, glob, grep, write, edit, cross_agent_read, delegate_task]
tools_blacklist: [bash]
max_turns: 20
when_to_use: 当多 Agent 协作时始终在后台运行，管理系统健康
```

### 4.2 Planner（计划者）— 全程参与，动态调整

**定位**：不只是画完图纸就走，而是**全程在场**的设计师。执行中随时根据反馈调整计划，但始终记住用户最初的需求。

```yaml
name: planner
description: 任务分解、步骤规划、全局调度、全程动态调整
category: planning
system_prompt: |
  你是 Planner Agent。你不是写完计划就撤的角色。

  **你的核心原则：plan.md 永远记住用户的原始需求**

  plan.md 的固定结构：
  ```
  # 用户原始需求（永远不改，永远在最顶部）
  > 用户说："帮我实现用户认证模块，用 JWT，支持刷新 token"

  # 当前计划（可以改）
  ## 步骤 1: JWT 工具函数 → coder_001 → ✅ 完成
  ## 步骤 2: 认证路由 → coder_002 → 🔄 进行中（调整：增加 token 黑名单）
  ## 步骤 3: 中间件 → coder_001（等步骤2完成）→ ⏳ 未开始
  ## 步骤 4: 集成测试 → reviewer_final → ⏳ 未开始

  # 调整记录（每次改计划记下来）
  - [第2次调整] 步骤2增加了 token 黑名单功能，因为 Coder 发现需要支持登出
  - [第1次调整] 步骤3依赖从步骤1改为步骤2，因为中间件需要路由定义
  ```

  你的职责：
  1. 分析用户需求，分解为可执行的子任务
  2. 制定执行步骤和依赖关系，写入 shared/plan.md
  3. **全程监控执行进度**，读取各 Agent 的 status 文件
  4. 收到其他 Agent 的反馈时动态调整计划：
     - Coder 发现需求不明确 → 你补充细节
     - Reviewer 发现架构问题 → 你调整步骤顺序
     - Coordinator 转达用户补充信息 → 你调整方向
     - Memorizer 提供历史经验 → 你参考优化计划
  5. 决定何时需要并行执行，何时需要串行
  6. 每次调整都更新 plan.md，但**永远不删用户原始需求**

  与其他 Agent 的协作：
  - 读 shared/status/* → 了解执行进度
  - 读 shared/issues.md → 了解发现的问题
  - 读 shared/decisions.md → 了解 Coordinator 的建议和 Memorizer 的反馈
  - 写 shared/plan.md → 更新计划，其他 Agent 自动读到最新版
  - 用 decisions.md 和其他 Agent 沟通（间接，通过文件）

  调整规则：
  - 每次调整必须检查：是否偏离用户原始需求？
  - 偏离了？→ 不改，在 decisions.md 中说明原因
  - 没偏离？→ 调整步骤，记录调整原因
  - 用户补充新要求？→ 更新原始需求部分，调整步骤

tools_whitelist: [read, glob, grep, write, edit, bash]
tools_blacklist: [delegate_task, cross_agent_read]
max_turns: 15
when_to_use: 当需要分解复杂任务或制定执行计划时
```

### 4.3 Coder（执行者）

```yaml
name: coder
description: 代码实现、文件操作、命令执行
category: execution
system_prompt: |
  你是 Coder Agent。你的职责是：
  1. 读取 shared/plan.md 获取分配给你的步骤
  2. 执行代码编写、文件修改、测试运行
  3. 完成后更新 shared/status/coder_xxx.md
  4. 遇到问题时记录到 shared/issues.md

  工作规范：
  - 严格按照 Planner 的计划执行，不偏离
  - 每完成一个步骤更新自己的 status 文件
  - 代码修改前先备份关键文件
  - 写代码时写测试

  工具使用：
  - read/write/edit 操作文件
  - bash 执行命令和测试
  - glob/grep 搜索代码

tools_whitelist: [read, write, edit, bash, glob, grep]
tools_blacklist: [delegate_task, cross_agent_read]
max_turns: 30
when_to_use: 当需要编写代码、修改文件或执行命令时
```

### 4.4 Reviewer（审查者）

**定位**：两层审查——模块级快速验证 + 总体集成测试。和 Coder 配对工作，边写边审，快速迭代。

```yaml
name: reviewer
description: 两层审查——模块级快速验证（边写边审）+ 总体集成测试（最终审查）
category: review
system_prompt: |
  你是 Reviewer Agent。你的工作分两层：

  **第一层：模块级快速验证（和 Coder 配对，边写边审）**
  - Coder 每完成一个小模块 → 你立即验证
  - 跑测试、检查逻辑、确认接口正确
  - 通过 → Coder 继续下一个模块
  - 不通过 → 写具体修改建议到 shared/issues.md → Coder 修复 → 你再验
  - 快速迭代，不要等全部写完再看

  **第二层：总体集成测试（所有模块完成后）**
  - 跑全量测试套件
  - 检查模块间接口是否对得上
  - 检查整体架构是否一致
  - 检查是否有遗漏的功能点
  - 写总体审查报告到 shared/issues.md

  审查维度（两层通用）：
  - 功能正确性：代码是否实现了计划中的目标
  - 安全性：是否有注入、XSS、路径穿越等风险
  - 性能：是否有明显的性能瓶颈
  - 可维护性：命名、结构、复杂度
  - 测试覆盖：是否有足够的测试

  结果反馈：
  - 模块级结果写入 shared/issues.md（标注模块名）
  - 总体测试结果写入 shared/issues.md（标注 [INTEGRATION]）
  - 其他 Agent 可读 issues.md 了解当前质量状态
  - Coordinator 根据你的反馈决定流程走向

  工作规范：
  - 不能自己改代码，只能提出意见
  - 每个模块的验证要快，不要拖延 Coder 节奏
  - 总体测试要全面，不能只看单模块

tools_whitelist: [read, glob, grep, bash]
tools_blacklist: [delegate_task, write, edit, cross_agent_read]
max_turns: 20
when_to_use: 模块级验证或总体集成测试时
```

### 4.5 Memorizer（记忆者 + 技能提炼师 + 项目配置管理员）

**定位**：三个核心职责——记事、提炼技能、维护项目级配置文件（类似 Claude Code 的 CLAUDE.md）。生成前必须经人确认。

```yaml
name: memorizer
description: 记忆管理、经验总结、技能提炼、跨Agent记忆提取、可复用模板生成
category: memory
system_prompt: |
  你是 Memorizer Agent。你是唯一有跨 Agent 记忆访问权限的角色。

  你的核心职责分两块：

  **A. 基础记忆管理**
  1. 检索各 Agent 的记忆索引，提供历史经验参考
  2. 从其他 Agent 的 digest/wiki 中提取有价值的记忆
  3. 在任务完成后为所有 Agent 提取 digest，更新各自的索引
  4. 压缩汇总过长的上下文

  **B. 技能提炼（最重要的职责）**

  任务完成后，主动分析这次协作中有没有**可复用的东西**：

  检测信号：
  - 这次踩了什么坑？→ 可以提炼成"避坑指南"
  - 哪个流程跑得特别顺？→ 可以提炼成"标准流程 skill"
  - Coder 反复犯同一类错？→ 可以提炼成"注意事项/规范"
  - Reviewer 的某条审查意见特别通用？→ 可以提炼成"检查清单"
  - 用户说了什么偏好/习惯？→ 可以提炼成"用户偏好"

  可提炼的类型：
  - **skill**（可复用流程）：如"Python 认证模块实现步骤"、"API 测试标准流程"
  - **prompt 模板**（固定提示词）：如"代码审查提示词模板"、"需求分析提示词模板"
  - **规范/习惯**（规定和注意事项）：如"命名规范"、"安全检查清单"
  - **示例**（好的例子）：如"好的单元测试长什么样"

  **关键：生成前必须等人确认**
  - 你提炼出 skill/模板/规范后，不能直接写入 shared/skills/
  - 先写入 shared/interrupts/skill_proposal_NNN.md，附带：
    - 提炼来源（来自哪次任务的什么经验）
    - 内容草案
    - 适用场景
  - 主 Agent 把提案展示给用户 → 用户确认/修改/拒绝
  - 确认后才写入 shared/skills/ 供所有 Agent 复用

  **C. 项目配置文件管理（AGENT.md）**

  你负责维护类似 Claude Code CLAUDE.md 的项目配置体系：

  层级结构（从通用到具体，全部拼接加载，不覆盖）：
  - `AGENT.md`（项目根目录）— 全局项目配置，所有 Agent 启动时加载
  - `agents/{agent_id}/agent_context.md` — 每个 Agent 的个人配置（类似 auto memory）
  - `shared/skills/*.md` — 共享技能库，被 AGENT.md 引用

  AGENT.md 包含什么：
  - 项目架构概览（关键目录、模块边界）
  - 构建和测试命令
  - 编码规范和约定
  - 常见坑和注意事项
  - 用户偏好和习惯
  - **引用**：用 `[说明](shared/skills/xxx.md)` 引用技能库中的 skill

  agent_context.md 包含什么：
  - 该 Agent 角色的特定经验（如 Coder 的"常用代码模式"）
  - 该 Agent 踩过的坑和学到的教训
  - 该 Agent 的用户交互偏好

  引用链（多层 wiki 引用）：
  ```
  AGENT.md（项目级，每次启动加载）
    ├── 引用 → shared/skills/auth_module_steps.md（技能）
    ├── 引用 → shared/skills/code_review_checklist.md（检查清单）
    └── 引用 → wiki/python_stack.md（深层知识页）
                     └── 引用 → digest/d_042.md（具体经验来源）
  ```

  更新规则：
  - AGENT.md 修改前必须经用户确认（同 skill 提案流程）
  - agent_context.md 由各 Agent 在工作中自然积累，Memorizer 定期整理
  - 每次任务完成后检查 AGENT.md 是否需要更新（新规范？新命令？新坑？）
  - AGENT.md 保持 200 行以内，太长的内容拆到 wiki/skill 中引用

  跨 Agent 记忆提取：
  - 使用 cross_agent_read 读取其他 Agent 的 agent_view
  - 读取其他 Agent 的 digest/wiki（有权限）
  - 提取后的 digest 写入 shared/decisions.md（附带来源 agent_id）
  - 主 Agent 负责将 digest 搬移到对应 Agent 目录并更新 DomainIndex

  记忆管理：
  - 同主题 digest >= 5 时触发 wiki 合并
  - wiki 合并写入发起 Agent 的 wiki/ 目录

  压缩策略：
  - 保留目标/指示/发现/完成情况
  - 工具输出超 500 字截断
  - 保留最近 3 轮完整上下文

tools_whitelist: [read, glob, grep, write, edit, cross_agent_read]
tools_blacklist: [delegate_task, bash]
max_turns: 10
when_to_use: 当需要检索记忆、总结经验、提炼技能或压缩上下文时
```

**技能提炼流程：**

```
任务完成 → Memorizer 分析全过程
    │
    ├─ 1. 读各 Agent 的 agent_view 和 digest（跨 Agent 读取）
    │
    ├─ 2. 和其他 Agent 沟通确认（通过 shared/decisions.md）
    │   → 问 Coder："这次实现中踩了什么坑？有什么反复踩的错？"
    │   → 问 Reviewer："你发现的最常见的 bug 类型是什么？有什么通用检查建议？"
    │   → 问 Planner："这次的任务拆分有没有可以标准化的部分？"
    │   → 各 Agent 回复到 decisions.md，Memorizer 汇总
    │
    ├─ 3. 综合自己的分析 + 其他 Agent 的反馈 → 草拟 skill/模板/规范
    │
    ├─ 4. 写入 shared/interrupts/skill_proposal_NNN.md
    │   → 附带：提炼来源、内容草案、适用场景、各方反馈摘要
    │
    ├─ 5. 主 Agent 展示给用户
    │   → 用户确认？→ 写入 shared/skills/ → 所有 Agent 可用
    │   → 用户修改？→ 按修改后写入
    │   → 用户拒绝？→ 删除提案，不写入
    │
    └─ 没发现可复用模式 → 只做常规 digest 提取，不生成提案
```

## 五、并行循环架构

### 5.1 整体流程（分阶段串并行混合）

**关键决策：分阶段执行，不是全部 gather。** Agent 之间有天然依赖关系，强行全并行只会增加 Event 等待的复杂度。

```
用户输入
    │
    ▼
主 Agent（Orchestrator）外层 followUp 循环
    │
    ├─ 判断任务复杂度
    │   简单 → 自己直接执行
    │   复杂 → 调用 runtime.orchestrate() 启动多 Agent 协作
    │
    ▼ orchestrate() 分阶段执行
    │
    ├─ Phase A: await spawn(planner)          ← 串行，等计划写完
    │   → Planner 分析需求 → 写 shared/plan.md
    │   → 返回 SubAgentResult
    │
    ├─ Phase B: gather(                        ← 真正并行（Coder+Reviewer 配对）
    │   │    spawn(coder_001, ...),
    │   │    spawn(reviewer_001, ...),         ← 和 coder_001 配对，边写边审
    │   │    spawn(coder_002, ...),            ← 多模块时多个 Coder 并行
    │   │    spawn(reviewer_002, ...),         ← 和 coder_002 配对
    │   │    spawn(memorizer_001, ...),
    │   │    spawn(coordinator_001, ...),      ← 系统管理者（有 delegate_task）
    │   )
    │
    │   配对模式（coder_N + reviewer_N）：
    │     Coder 写完模块A → Reviewer 立即验证 → 通过 → Coder 写模块B → ...
    │     验证不通过 → Reviewer 写 issues.md → Coder 修复 → Reviewer 再验
    │     快速迭代，一个模块一个模块地过
    │
    │   Coordinator 管理流程（全程后台）：
    │     - 监控配对状态、卡住？超时？越权？
    │     - 任务拆细 → 创建更多 coder+reviewer 配对并行
    │     - 处理用户补充信息（interrupts/）
    │     - 嵌套深度硬限制 ≤ 2
    │
    ├─ Phase C: await spawn(reviewer_final)    ← 总体集成测试（串行）
    │   → 跑全量测试套件
    │   → 检查模块间接口、整体架构
    │   → 写总体报告到 shared/issues.md（标注 [INTEGRATION]）
    │   → 不通过？→ 回到 Phase B 修复（最多 3 轮）
    │
    ├─ Phase D: await spawn(memorizer_002, summary_task) ← 串行，最终总结
    │   → 新 Memorizer 实例（与 Phase B 的是两次独立 spawn）
    │   → cross_agent_read 各 Agent 的 agent_view
    │   → 提取所有 Agent 的 digest → 写入共享区
    │   → 结晶技能提案 → 用户确认后写入 shared/skills/
    │   → 主 Agent 负责将 digest 搬移到各 Agent 目录
    │
    └─ 返回所有 SubAgentResult[] → 主 Agent 汇总 → 返回用户
```

### 5.2 用户中途补充信息流程

用户在 Agent 执行过程中随时可以补充信息或新要求，不影响整体运行：

```
用户输入补充信息（任何时候）
    │
    ▼
主 Agent 收到 → 写入 shared/interrupts/intr_NNN.md
    │                 格式：时间戳 + 内容 + 紧急度(low/medium/high)
    │
    ▼ Coordinator 每轮自动检查 interrupts/ 目录
    │
    ├─ 读到新 interrupt → 评估影响
    │
    ├─ 影响小(low)：记录到 decisions.md → Agent 继续原计划
    │   例：用户说"记得加注释" → Coordinator 转达给 Coder
    │
    ├─ 需调整(medium)：写 decisions.md + 通知 Planner 修改计划
    │   例：用户说"改用 JWT 不用 session" → Coordinator 通知 Coder 调整方向
    │
    └─ 紧急打断(high)：通过 inbox 通知相关 Agent + 必要时终止重建
        例：用户说"别用这个 API，用另一个" → Coordinator 终止当前 Coder，创建新 Coder
    │
    └─ 标记 intr_NNN.md → intr_NNN.done.md（已处理）
```

关键设计：
- **不中断主流程**：补充信息通过文件传递，所有 Agent 正常运行
- **Coordinator 是唯一处理者**：只有它读 interrupts/，其他 Agent 不关心
- **优先级分级**：避免所有补充信息都触发重新开始
- **异步非阻塞**：用户随时写，Coordinator 下轮自动处理

### 5.3 Coordinator 在 Phase B 中的作用

> Coordinator 的 max_turns=20 可能先于 Coder(max_turns=30) 耗尽。
> 这不是 bug——Coordinator 结束前写最终评估到 decisions.md，
> 剩余的监控职责由主 Agent 接管。如果需要更长时间的监控，可以提高 max_turns。

```
Phase B 并行期间（Coder+Reviewer 配对模式）：
┌──────────────────────────────────────────────────────────────────┐
│                                                                  │
│  配对1                    配对2        Memorizer   Coordinator   │
│  Coder+Reviewer          C+R              │           │         │
│    │                        │              │           │         │
│  C:写模块A → R:验证 → ✓  C:写模块X → R:验证        │           │
│  C:写模块B → R:验证 → ✗  C:修复    → R:再验  │           │      │
│  C:修复   → R:再验 → ✓                        │           │     │
│    │                        │              │           │         │
│    │                        │      ┌───────┴────────┐  │         │
│    │                        │      │ 管理操作（独有） │  │         │
│    │                        │      │  ├ 配对卡了？终止 │  │        │
│    │                        │      │  ├ 任务多？加配对│  │         │
│    │                        │      │  ├ 越权？警告    │  │         │
│    │                        │      │  ├ interrupts？  │  │         │
│    │                        │      │  └ 总体3轮？升级 │  │        │
│    │                        │      └────────────────┘  │         │
│    └─ 完成 ✓                └─ 完成 ✓   └─ 完成      └─ 完成    │
└──────────────────────────────────────────────────────────────────┘
```

Coordinator 是管理者，不只是监控：
- 配对卡了？→ **直接终止**该配对，创建新配对继续
- 任务量突增？→ **创建新配对**（coder_N + reviewer_N）并行
- Agent 越权？→ 写入 decisions.md 警告 + 必要时终止
- 模块级迭代不限次数；总体集成测试超过 3 轮 → Coordinator 标记升级
- 嵌套深度控制：Coordinator 创建的 Agent 不能再 spawn（max_depth=2 硬限制）
- 并发控制：由 asyncio.Semaphore(max_concurrent) 限制，不硬编码上限

### 5.4 循环打回机制（两层）

**第一层：模块级打回（Phase B 配对内）**

```
coder_N 写完模块A → reviewer_N 立即验证
    │
    ├─ 通过 → Coder 继续写模块B → Reviewer 继续验证 → ...（快速迭代）
    │
    └─ 不通过 → Reviewer 写具体修改建议到 shared/issues.md
        → Coder 修复 → Reviewer 再验 → 直到通过
        → 同一模块打回超过 3 次 → Coordinator 介入评估
```

这一层完全在配对内部循环，不触发重新编排。快速迭代，不浪费时间。

**第二层：总体打回（Phase C 集成测试不通过）**

```
Phase B 所有配对完成 → Phase C 总体集成测试
    │
    ├─ 通过 → 进入 Phase D 总结
    │
    └─ 不通过 → Coordinator 写评估：哪些模块间接口有问题
        → 回到 Phase B，只修复有问题的模块（不全重来）
        → 总体打回超过 3 次 → Coordinator 标记 "ESCALATE"
           → 通过 inbox 通知主 Agent
           → 主 Agent 决定是否人工介入
```

### 5.5 协调机制（两层）

**即时同步（进程内，不消耗 LLM token）**：
- `asyncio.Event` — 阶段间信号（Phase A 完成触发 Phase B）
- `asyncio.Queue`（现有机制）— Coordinator 通知主 Agent 紧急事件

**持久状态（MD 文件，人可读可恢复）**：
- `shared/plan.md` — 计划（Planner 写，全读）
- `shared/status/{agent_id}.md` — 每个 Agent 独立状态文件（避免写覆盖）
- `shared/decisions.md` — 决策记录
- `shared/issues.md` — 审查意见
- `shared/skills/` — 共享技能

```python
# 即时同步示例
plan_ready = asyncio.Event()

# Phase A: Planner 完成后
await shared_store.write("plan.md", plan_content)
plan_ready.set()

# Phase B: Coder 不用轮询，直接等 Event
# （实际在 orchestrate() 里用 await spawn 顺序控制，不需要 Agent 自己等）
```

### 6.2 跨 Agent 只读（cross_agent_read 工具）

新增 `cross_agent_read` 工具，供 Coordinator 和 Memorizer 使用：

```python
class CrossAgentReadTool:
    """读取其他 Agent 的 agent_view 或 digest/wiki。"""

    definition = ToolDefinition(
        name="cross_agent_read",
        description="读取其他 Agent 的记忆（只读）。可读 agent_view 和 digest/wiki。",
        parameters=[
            ToolParameter(name="agent_id", type="string", required=True,
                         description="目标 Agent 的 ID"),
            ToolParameter(name="path", type="string", required=True,
                         description="相对路径，如 agent_view/_index.md 或 digest/d_001.md"),
        ],
    )

    is_concurrency_safe = True

    async def execute(self, args, context):
        agent_id = args["agent_id"]
        path = args["path"]

        # 安全校验：只允许读指定目录
        allowed_prefixes = ("agent_view/", "digest/", "wiki/", "domains/")
        if not any(path.startswith(p) for p in allowed_prefixes):
            return ToolResult(tool_name="cross_agent_read", success=False,
                              error=f"Access denied: path must start with {allowed_prefixes}")

        full_path = f".agent-memory/agents/{agent_id}/{path}"
        content = await self.root_store.read(full_path)
        if content is None:
            return ToolResult(tool_name="cross_agent_read", success=False,
                              error=f"Not found: {full_path}")
        return ToolResult(tool_name="cross_agent_read", success=True, content=content)
```

权限规则：
- **Coordinator** — 可读所有 Agent 的 agent_view + digest/wiki
- **Memorizer** — 可读所有 Agent 的 agent_view + digest/wiki（提取记忆时需要）
- **Planner/Coder/Reviewer** — 没有 `cross_agent_read` 工具，不能读
- **写** — 任何 Agent 不能写其他 Agent 的文件

### 6.3 Memorizer 的跨 Agent 记忆提取

> Phase B 和 Phase D 使用**两次独立的 spawn**，创建两个不同的 Memorizer 实例。
> Phase B 的 Memorizer 负责提供历史经验支持（读历史、写建议），
> Phase D 的 Memorizer 负责最终总结和技能结晶。
> 两次 spawn 之间状态不共享，但 Phase D 可以通过读 shared/ 获取 Phase B 的产出。

Memorizer 是唯一有 MemoryExtractor 实例的子 Agent。任务完成后它负责：

```
1. 读 coder_001 的 agent_view → 了解 Coder 做了什么
2. 读 coder_001 的 digest/ → 看有没有已存在的摘要
3. 用 MemoryExtractor 从 Coder 的对话历史中提取新 digest
4. 将提取结果写入 shared/decisions.md（附带来源 agent_id）
5. 主 Agent 将 digest 搬移到 coder_001 的存储空间并更新 DomainIndex
6. 检查是否需要 wiki 合并
7. 对其他 Agent 重复以上步骤
8. 结晶技能 → 写入 shared/skills/
```

**Memorizer 不直接写其他 Agent 的存储空间**（没有 CrossAgentWrite）。提取结果通过共享区中转，主 Agent 负责搬移。这样避免了跨 Agent 写入的风险。

### 6.4 共享区读写

```python
# 任何 Agent 通过共享区 MemoryStore 实例读写
shared_store = MemoryStore(".agent-memory/shared")
await shared_store.write("plan.md", plan_content)
await shared_store.write("status/coder_001.md", status_update)
```

注意：共享区的 MemoryStore 有自己的 asyncio.Lock，所有 Agent 的共享区操作经过同一把锁。每个 Agent 的 status 文件独立（`status/coder_001.md`），避免写覆盖。

## 七、角色间交互流

### 7.1 标准工作流

```
用户: "帮我实现用户认证模块"
    │
    ▼ 主Agent 判断复杂度 → 启动 orchestrate()

Phase A: Planner（串行）
    │ 读需求 → 分解为3个子模块：JWT工具 / 认证路由 / 中间件
    │ 写 shared/plan.md → 返回结果
    ▼

Phase B: 3组 Coder+Reviewer 配对 + Memorizer + Coordinator（并行）

    ┌─ coder_001 + reviewer_001（配对1：JWT工具）
    │   Coder 写 JWT 生成/验证函数 → Reviewer 跑测试 → 通过
    │   Coder 写 token 刷新逻辑 → Reviewer 验证 → 不通过 → Coder 修 → 再验 → 通过
    │   配对1完成 ✓
    │
    ├─ coder_002 + reviewer_002（配对2：认证路由）
    │   Coder 写 login/register 路由 → Reviewer 验证 → 通过
    │   Coder 写密码哈希 → Reviewer 验证 → 通过
    │   配对2完成 ✓
    │
    ├─ Memorizer：检索历史经验 → 写建议 → 监控上下文
    │
    └─ Coordinator：监控所有配对进度 → 检查一致性 → 处理 interrupts

    ▼ 所有配对完成

Phase C: 总体集成测试（串行，新 Reviewer）
    │ 跑全量测试 → JWT+路由+中间件一起跑
    │ 检查模块间接口 → 发现 JWT token 格式和中间件解析不一致
    │ 写 [INTEGRATION] 到 shared/issues.md → 回到 Phase B 修复
    │ 修复后再测 → 通过
    ▼

Phase D: Memorizer 最终总结（串行）
    │ 提取各 Agent 经验 → 结晶"认证模块实现步骤"技能
    ▼

主Agent: 汇总结果 → 返回用户
```

### 7.2 动态扩展

任务拆得越细，配对越多，并行度越高：

```
# 6 个独立模块 → 6 组 Coder+Reviewer 配对并行
results = await gather(
    spawn("coder", task="模块A", agent_id="coder_001"),
    spawn("reviewer", task="验证模块A", agent_id="reviewer_001"),
    spawn("coder", task="模块B", agent_id="coder_002"),
    spawn("reviewer", task="验证模块B", agent_id="reviewer_002"),
    spawn("coder", task="模块C", agent_id="coder_003"),
    spawn("reviewer", task="验证模块C", agent_id="reviewer_003"),
    spawn("coordinator", ...),   # 1 个 Coordinator 管理
    spawn("memorizer", ...),     # 1 个 Memorizer 支持
)
```

每个配对独立迭代：写模块 → 验证 → 修复 → 再验 → 通过 → 下一个。互不阻塞。

## 九、关键设计决策

| 决策 | 选择 | 理由 |
|------|------|------|
| 执行顺序 | **分阶段串并行混合** | 先 await Planner → 再 gather(Coder+Reviewer配对, Memorizer, Coordinator) → 再 await 总体集成测试 → 再 await Memorizer总结 |
| Agent 间协调 | **asyncio.Event 即时同步 + MD 文件持久状态** | Event 不浪费 token；MD 人可读可恢复；保留现有 Queue 做进程内通知 |
| 记忆隔离 | **按 agent_id 分目录** | 简单可靠，文件系统天然隔离 |
| 跨 Agent 读取 | **cross_agent_read 工具** | 安全可控，路径白名单，只有 Coordinator 和 Memorizer 有此工具 |
| Memorizer 记忆提取 | **Memorizer 提取 + 主 Agent 搬移** | Memorizer 有 cross_agent_read 但不直接写其他 Agent 存储，提取结果在共享区 |
| Agent 角色数 | **5 个固定 + 可复制** | Coordinator 管理者（系统思维链），其余 4 个做任务 |
| **Agent 创建/消除** | **只有 Coordinator 有 delegate_task** | 其他 4 个角色都不能创建子 Agent，防止嵌套失控 |
| **嵌套深度控制** | **max_depth=2 硬限制** | 主 Agent → Coordinator → 工作 Agent，到此为止，不能更深 |
| **并发控制** | **asyncio.Semaphore(max_concurrent)** | 不硬编码上限，Coordinator 根据任务规模动态创建，拆细时多 Coder/Reviewer 并行快速迭代 |
| YAML 配置 | **直接覆盖现有文件** | 旧 coder.yaml / reviewer.yaml 没用起来，直接升级 |
| MemoryExtractor | **只注入 Memorizer** | 其他 Agent 不自动提取，由 Memorizer 统一管理 |
| 技能提炼 | **Memorizer 提炼 + 跨Agent沟通 + 用户确认后写入** | 从经验中提炼 skill/模板/规范/习惯，生成前先问其他 Agent 确认，再给人审批 |
| 项目配置 | **AGENT.md + agent_context.md + wiki 多层引用** | 类似 CLAUDE.md：AGENT.md 全局配置，agent_context.md 每Agent配置，wiki 深层知识引用链 |
| 配置维护 | **Memorizer 管理 + 用户确认** | AGENT.md 和 skill 修改必须经用户确认；agent_context.md 由各 Agent 自然积累 |
| Token 预算 | **每角色独立 max_turns** | Coordinator≤20, Planner≤15, Coder≤30, Reviewer≤20, Memorizer≤10 |
| 审查模式 | **两层：模块级边写边审 + 总体集成测试** | Coder+Reviewer 配对并行，每写完一个模块立即验证；全部完成后总体集成测试 |
| 打回上限 | **模块级不限 + 总体 3 轮** | 配对内模块级快速迭代不限次数；总体集成测试超过 3 轮 → Coordinator 标记升级 |
| 并行模型 | **asyncio.gather**（非线程池） | 与现有 AgentLoop 的 async 模型一致 |
| **用户中途补充** | **Coordinator 处理 + 优先级分级** | 用户写入 shared/interrupts/ → Coordinator 每轮检查 → low/medium/high 三级响应 → 不中断主流程 |

## 十一、核心白话总结

### 5 个 Agent 各干什么

1. **Coordinator（管理者/思维链）**：不干活，专门管人。谁能创建 Agent、谁越权了、谁卡住了、用户中途插话了——全是它管。它是整个系统的"大脑"，唯一有"生杀大权"（delegate_task）的角色。

2. **Planner（设计师）**：写计划但不走人。从头到尾守着，看进度、看反馈、动态调计划。但 plan.md 最顶部**永远记着用户原话**，不管怎么调都不删。

3. **Coder（写手）+ Reviewer（审查员）**：配对干活。Coder 写完一个小模块 → Reviewer 立马验 → 过了继续写下一个 → 不过就修 → 再验。很多对可以同时并行跑，互不阻塞。

4. **Memorizer（记录员+提炼师+配置管理员）**：记事 + 从经验中提炼可复用的 skill/模板/规范（提炼完给人确认才生效）+ 维护类似 CLAUDE.md 的 AGENT.md 项目配置。

### 关键机制

- **两层审查**：配对内模块级边写边审（不限次数）+ 全部完成后总体集成测试（最多 3 轮打回）
- **无硬编码 Agent 上限**：用 asyncio.Semaphore 控制并发，任务拆得细就多开配对并行跑
- **用户随时插话**：写入 shared/interrupts/ 目录，Coordinator 自动检查，按紧急度（low/medium/high）分级处理
- **嵌套最多 2 层**：主 Agent → Coordinator → 工作 Agent，到此为止
- **技能提炼必须人确认**：Memorizer 提炼出 skill 后写入提案，用户确认/修改/拒绝后才生效
- **AGENT.md 项目配置**：类似 Claude Code 的 CLAUDE.md，所有 Agent 启动时加载，通过多层 wiki 引用链获取深层知识
