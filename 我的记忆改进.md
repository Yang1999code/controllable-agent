# 记忆系统改进方案

> 基于 Synthius-Mem 论文 + OpenCode 指针映射 + 卡帕西 Wiki 层级思想
> 核心机制：任务单元后批量提取 → digest摘要 → 积累合并wiki知识页
> 存储格式：Markdown + YAML frontmatter（人可读 + 机器可查）
> 提取者：专门轻量模型（与主Agent分离）

---

## 一、核心思想

**三个关键设计**：

1. **Wiki式层级提取**：不是每轮对话都提取，而是等一个任务单元完成后批量处理。先生成 digest 摘要（标题+要点），积累多个同主题 digest 后合并为 wiki 知识页。大部分信息这两层就够了。

2. **事实级指针去重**：对话产生的事实只存一份，四个域各存指针。无论多少Agent参与，事实不重复。

3. **抗幻觉**（来自论文）：只存"确实说过"的事实。检索时查不到 → 空=没说过 → 不编造。类似卡帕西 Wiki：有据可查才能说。

```
任务单元完成
    │
    ▼
轻量模型提取 → digest 摘要（标题+要点）
    │
    ▼ (积累N个digest)
    │
    ▼
轻量模型合并 → wiki 知识页（完整知识页）
    │
    ┌───────────┼───────────┐
    ▼           ▼           ▼
 对话域      个人域      Agent域
 (指针)      (指针)      (指针)
```

## 二、提取策略（最关键的设计）

### 2.1 什么时候提取？

**Agent自动判断任务单元完成时触发**。

任务单元 = 几次对话解决了一个具体问题。比如：
- "检查agent有没有问题" → 检查完 = 一个任务单元
- "做个记忆改进方案" → 方案写完 = 一个任务单元
- "修复这个bug" → bug修完 = 一个任务单元

Agent通过以下信号判断任务完成：
- 用户说"好了/谢谢/下一个"
- Agent完成了当前目标（无待办工具调用）
- 意图明显切换到新话题

### 2.2 谁来提取？

**专门的轻量模型**（和主Agent分离）。

好处：
- 主Agent不分心，专注执行任务
- 提取模型可以用更便宜的模型（如 Haiku 级别）
- 提取 prompt 可以专门优化，和主任务 prompt 互不干扰
- KV Cache 更友好（提取 prompt 固定，缓存命中率高）

### 2.3 Wiki式层级结构（digest → wiki）

> **命名说明**：用 `digest`（摘要）和 `wiki`（知识页）而非 L1/L2，避免和现有代码的 L0-L4 层级语义冲突。

**digest 摘要**（每次任务单元后生成）：

```markdown
---
id: d_001
level: digest
source_session: ses_abc123
task_summary: 检查controllable-agent项目质量
timestamp: "2026-05-03T14:30:00"
domains: [conversation, agent_view]
tags: [项目检查, 测试, 打包]
---

## 要点
- 项目有255个测试全部通过
- pyproject.toml打包配置有误（已修复）
- pytest-asyncio未安装（已修复）
- 代码质量良好，架构三层分离清晰
```

**wiki 知识页**（积累5+个相关 digest 后合并）：

```markdown
---
id: w_python_stack
level: wiki
source_digests: [d_001, d_005, d_012]
updated: "2026-05-03T15:00:00"
domains: [profile, agent_view]
tags: [python, 技术栈, 偏好]
---

## Python 技术栈

### 编程语言
- 偏好 Python 3.12（来源: d_005）
- 对 TypeScript 较新但正在学习（来源: d_012）

### 框架
- 项目使用自研Agent框架 controllable-agent（来源: d_001）
- 三层架构: ai/ → agent/ → app/（来源: d_001）

### 工具链
- 使用 Claude Code + 多个插件（来源: d_005）
- 代理: Clash Verge 端口 7897（来源: d_008）

### 决策记录
```

### 2.4 层级规则

| 层级 | 何时生成 | 内容 | 粒度 |
|------|---------|------|------|
| **digest 摘要** | 每个任务单元完成后 | 标题 + 3-5个要点 + 标签 | 一个任务的产出 |
| **wiki 知识页** | 积累5+个同主题digest后 | 完整知识页，带来源标注 | 一个主题的全貌 |
| **overview 综述** | 极少用，跨多个wiki | 如"用户全貌"这种顶层文档 | 一个域的综述 |

**大部分情况 digest + wiki 两层就够了**。overview 只在极端复杂的长期项目才需要。

## 五、检索机制

### 5.1 检索优先级

```
用户查询
    │
    ▼
1. 先查 wiki（最完整，命中率最高）
    │ 没命中
    ▼
2. 再查 digest（更细粒度）
    │ 没命中
    ▼
3. 没找到 → 空=没说过 → 不编造（抗幻觉）
```

### 5.2 全局索引

```markdown
---
type: keyword_index
updated: "2026-05-03T15:00:00"
---

## python
- wiki: [Python技术栈](wiki/python_stack.md)
- digest: [d_001](digest/d_001.md), [d_005](digest/d_005.md)

## 测试
- wiki: [项目质量](wiki/project_quality.md)
- digest: [d_001](digest/d_001.md)

## 偏好
- wiki: [Python技术栈](wiki/python_stack.md), [开发环境](wiki/dev_environment.md)
- digest: [d_012](digest/d_012.md)
```

### 5.3 检索方法

| 方法 | 用途 | 查什么 |
|------|------|--------|
| **关键词搜索** | 日常查找 | jieba分词 → 查全局索引 → 返回匹配的digest/wiki |
| **域内浏览** | 按场景查看 | 读取 domains/{域}/_index.md → 列出该域所有条目 |
| **时间范围** | 回顾历史 | 遍历digest的timestamp字段，筛选时间区间 |
| **精确ID查找** | 指针解析 | 根据域文件中的pointer字段直接定位digest/wiki文件 |

## 六、完整工作流

```
对话进行中
    │
    ├─ Agent正常工作（主模型）
    │
    ├─ Agent判断: 这个任务单元完成了
    │   信号: 用户确认 / 目标达成 / 意图切换
    │
    ▼
触发 digest 提取（轻量模型）
    │
    ├─ 输入: 本任务单元的对话历史
    ├─ 输出: digest 摘要文件 → 写入 digest/d_xxx.md
    ├─ 更新: 域目录 + index.md
    ├─ 显示: "[记忆] 正在提取本次任务记忆..."
    │
    ▼
wiki 合并检查
    │
    ├─ 检查同主题 digest 数量 >= 5?
    │   是 → 触发 wiki 合并（轻量模型）
    │       输入: 所有同主题 digest
    │       输出: wiki 知识页 → 写入 wiki/{topic}.md
    │       更新: 域目录指向 wiki + index.md
    │   否 → 等待积累
    │
    ▼
去重检查（LLM判断）
    │
    ├─ 新 digest 与已有 wiki 内容对比
    ├─ 重复 → 跳过
    ├─ 冲突 → 更新 wiki（标注变更来源）
    ├─ 新增 → 写入
```

## 七、多Agent协作

```
主Agent完成子任务 → spawn 子Agent
    │
    ├─ 子Agent继承: 主Agent的agent_view文件
    │
    ├─ 子Agent独立工作
    │   完成后生成自己的digest摘要
    │   digest中标注 agent_id: "sub_agent_x"
    │
    └─ 子Agent完成 → 主Agent合并视角
       读取子Agent的digest → 更新自己的agent_view
       同一事实不重复，只是多了子Agent的评估角度
```

## 十、KV Cache 成本优化

提取模型调用设计为缓存友好：

| 设计 | 做法 | 效果 |
|------|------|------|
| 固定 system prompt | 提取指令永远不变 | 首次后每次命中缓存 |
| prefix 连续 | 对话历史不截断，保持连续 | 前文prefix命中 |
| 只追加新内容 | 新轮次追加到尾部 | prefix命中率>80% |
| 轻量模型 | 用Haiku级别模型 | 单次提取成本极低 |
| 批量而非逐轮 | 一个任务单元一次调用 | 调用次数少 |

预估：一个任务单元提取 ≈ 2000 input tokens（80%缓存命中） + 500 output tokens。成本可忽略。

## 十二、与现有代码适配性审查

### 12.1 发现的 4 个问题

**问题1：digest/wiki 命名和现有 L1/L2 冲突**（严重）

现有 `backend.py` 第96-106行已定义 L0-L4 层级，含义与方案完全不同：

| 层级 | 现有含义 | 方案原设计含义 |
|------|---------|-------------|
| L1 | 导航层（关键词→路径） | 对话摘要 |
| L2 | 事实库（环境事实） | Wiki合并页 |

**修复**：不改现有层级语义，方案使用新目录名：
- `digest/` — 方案的"摘要"（原L1概念）
- `wiki/` — 方案的"知识页"（原L2概念）
- 现有 L0-L4 保持不变，新旧系统并行

**问题2：AgentLoop 绑定单一模型**（中等）

`loop.py` 第93行 `self.provider = provider`，整个循环只有一个模型。方案需要专用轻量模型做提取，但不能走 AgentLoop 的 provider。

**修复**：在 `cli.py` 新建第二个 `IModelProvider` 实例（轻量模型），注入到独立的 `MemoryExtractor`。不侵入 AgentLoop，通过 LOOP_END Hook 的闭包捕获 extractor。

```
cli.py:
  main_provider = create_provider(...)       # 主模型
  extract_provider = create_provider(        # 轻量模型
      model="gpt-4o-mini", ...)
  extractor = MemoryExtractor(extract_provider)
  hooks.on(LOOP_END, extractor.try_extract)  # Hook闭包捕获
```

**问题3：子Agent没有记忆沉淀机制**（中等）

`runtime.py` 第127-130行明确规定子Agent**不触发记忆**。完成后只有简单文本拼接（`autonomous_memory.py` 第194行）。

**修复**：分阶段实现。
- Phase 1：只做主Agent记忆，子Agent不提取
- Phase 2：在 `SUBAGENT_STOP` Hook 中携带子Agent对话历史，供主Agent的 extractor 提取
- Phase 3：子Agent独立视角指针

**问题4：现有 search_keywords 是全文件扫描**（轻微）

`index.py` 第68行每次搜索都遍历所有 `.md` 文件。digest 文件多了性能会差。

**修复**：方案的 `index.md` 倒排索引直接替代全文件扫描。不再逐个读文件内容，只查索引。

| asyncio.run() 反模式 | `autonomous_memory.py` 第224行 | 在已运行的事件循环中调用会死锁。提取触发必须全路径 async |
| 全局 Lock 串行化 | `store.py` 第21行 | 多Agent同时写记忆会被 Lock 串行化。高并发时可能瓶颈 |
| 子Agent临时ID | `runtime.py` 第320行 | 子Agent的 agent_id 是 uuid4().hex[:8]，不可追溯。需要稳定命名 |

## 十三、LLM确认策略

**原则：LLM高置信直接执行，低置信才打扰人。**

### 置信度阈值

| 场景 | confidence >= 0.85 | confidence < 0.85 |
|------|-------------------|-------------------|
| 任务是否完成 | 自动提取 | 问人："这个任务完成了吗？" |
| digest主题归类 | 自动归入候选 | 问人："这算不算和XX同一主题？" |
| digest去重/冲突 | 自动合并或覆盖 | 问人："这两条记忆看起来重复，合并吗？" |
| 提取质量自评 | 直接写入 | 问人："我提取了这些要点，确认一下？" |

### 交互格式

```
# LLM高置信（不打扰人）
[记忆] 任务完成，正在提取... ✓ 已保存摘要

# LLM低置信（需要人确认）
[记忆] 不确定这个任务是否完成，提取吗？[y/n]
[记忆] "用户偏好FastAPI" 和现有记录 "使用Django" 冲突，更新？[y/n]
```

### 实现

`TaskDetector` 的判断方法返回 `(is_complete: bool, confidence: float)`：
- `confidence >= 0.85` → 直接触发提取
- `confidence < 0.85` → 通过 TUI 的确认提示询问用户
- one-shot 模式下（无交互）→ 低于阈值的跳过，对话结束时兜底提取

## 十五、架构审查总结

### 整体评价

**设计思路正确**，和现有代码的设计理念（三层架构、Hook扩展、Provider抽象）高度一致。最大的优势是：
- 通过 Hook 系统接入，不侵入主循环
- 通过独立 Provider 调用轻量模型，不侵入 AgentLoop
- MD + frontmatter 和现有文件系统存储完全兼容

### 需要特别注意的

1. **命名冲突**：用 `digest/` 和 `wiki/` 而不是 L1/L2，避免和现有层级混淆
2. **async 一致性**：提取触发链路必须全程 async，不能混用 `asyncio.run()`
3. **分阶段交付**：先做主Agent记忆（Phase 1），子Agent记忆放 Phase 3
4. **渐进替换**：新旧系统并行，通过迁移记录管理切换，不好用可回滚
