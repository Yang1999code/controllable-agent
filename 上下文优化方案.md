# my-agent 上下文管理与 KV Cache 优化方案

> 研究日期：2026-04-30
> 参考项目：OpenCode（指针索引压缩）、Claude Code（KV Cache 断点检测 + Token 预算管理）

---

## 一、现状诊断

当前 my-agent 的上下文管理有三个核心问题：

| 问题 | 现状 | 影响 |
|------|------|------|
| 上下文限制硬编码 | `AgentConfig.max_context_tokens = 128000`，写死在代码里 | 切模型要改代码；GPT-4o 实际 128K、DeepSeek 128K、Claude 200K/1M，硬编码要么浪费要么超限 |
| 无溢出检测和压缩 | `loop.py` 只管 append 消息，不检查 token 数 | 昨天测试时 431 页 PDF 读入后直接 95% 上下文占用，模型"在过程中卡住" |
| 无 KV Cache 感知 | 每轮 system prompt 可能动态变化，消息结构无优化 | Cache 命中率低，重复计算浪费 latency 和费用 |

## 二、改进一：上下文限制由模型决定（非硬编码）

### 2.1 参考方案

**OpenCode 的做法**（`overflow.ts`）——最简洁：

```typescript
// model.limit.context 和 model.limit.input 由 Provider 配置定义
export function usable(input: { cfg: Config.Info; model: Provider.Model }) {
  const context = input.model.limit.context   // 模型总窗口
  if (context === 0) return 0
  const reserved = input.cfg.compaction?.reserved
    ?? Math.min(20_000, maxOutputTokens(input.model))
  return input.model.limit.input
    ? Math.max(0, input.model.limit.input - reserved)
    : Math.max(0, context - maxOutputTokens(input.model))
}
```

**Claude Code 的做法**（`getContextWindowForModel()`）——5 级优先级：

1. `CLAUDE_CODE_MAX_CONTEXT_TOKENS` 环境变量覆盖
2. 模型名含 `[1m]` 后缀 → 1M tokens
3. `getModelCapability(model).max_input_tokens`
4. 1M beta header + 支持的模型
5. 兜底：200K

### 2.2 my-agent 落地设计

在 `IModelProvider` ABC 中新增一个属性方法，每种 Provider 实现自己的窗口上限：

```python
# ai/provider.py — IModelProvider ABC 新增
class IModelProvider(ABC):
    @property
    @abstractmethod
    def context_window(self) -> int:
        """模型的实际上下文窗口大小（总 token 数）。"""
        ...

    @property
    def max_output_tokens(self) -> int:
        """模型的最大输出 token 数，用于预留空间。默认 4096。"""
        return 4096

    @property
    def usable_context(self) -> int:
        """可用于对话历史的 token 数 = context_window - 输出预留。"""
        reserved = min(20000, self.max_output_tokens)
        return max(0, self.context_window - reserved)
```

各 Provider 实现：

```python
class OpenAICompatibleProvider(IModelProvider):
    # 模型窗口映射（可覆盖）
    _CONTEXT_WINDOWS = {
        "gpt-4o": 128000, "gpt-4o-mini": 128000,
        "gpt-4-turbo": 128000, "gpt-3.5-turbo": 16385,
        "deepseek-chat": 65536, "deepseek-reasoner": 65536,
    }

    @property
    def context_window(self) -> int:
        env_override = os.getenv("MY_AGENT_MAX_CONTEXT_TOKENS")
        if env_override:
            return int(env_override)
        return self._CONTEXT_WINDOWS.get(self.model, 128000)

class AnthropicProvider(IModelProvider):
    @property
    def context_window(self) -> int:
        env_override = os.getenv("MY_AGENT_MAX_CONTEXT_TOKENS")
        if env_override:
            return int(env_override)
        if "1m" in self.model.lower() or "[1m]" in self.model.lower():
            return 1_000_000
        return 200000  # Claude 默认 200K
```

AgentConfig 去掉硬编码的 `max_context_tokens`，改为从 provider 动态读取：

```python
# AgentLoop 中
effective_limit = self.provider.usable_context  # 不再用 config.max_context_tokens
```

## 三、改进二：OpenCode 式指针索引压缩

### 3.1 核心概念：tail_start_id

OpenCode 的压缩不是"删旧留新"，而是**指针标记**：

```
压缩前消息流：
[msg1(user)] [msg2(asst)] [msg3(user)] [msg4(asst+tools)] [msg5(user)] [msg6(asst)]

select() 计算后：
  head = [msg1, msg2, msg3, msg4]  ← 这些将被摘要替代
  tail_start_id = msg5.id           ← 指针！指向保留段的第一条 user 消息
  tail = [msg5, msg6]               ← 这些完整保留

压缩后消息流：
  [summary_msg(asst, summary:true)] [msg5(user)] [msg6(asst)]
```

**关键优势**：`tail_start_id` 只是一个 message ID 字符串，存在 `compactionPart` 中。不需要复制消息、不需要移动数据。后续构建 API 请求时，通过这个 ID 找到保留段的起点。

### 3.2 三层压缩策略

OpenCode 和 CCB 都采用**递进式压缩**，不是一步到位：

```
第一层：Prune（无 API 调用）
  ↓ 不够
第二层：Compaction（LLM 摘要）
  ↓ 还不够
第三层：Emergency Truncation（直接截断）
```

#### 第一层：Prune — 清除旧工具输出

参考 `compaction.ts:173-218`：

```python
PRUNE_MINIMUM = 20_000   # 至少释放 20K token 才执行
PRUNE_PROTECT = 40_000   # 保留最近 40K token 的工具输出
PRUNE_PROTECTED_TOOLS = {"skill"}  # 这些工具的输出永远不清理

def prune(messages, model):
    """从后向前扫描，清除超过 PRUNE_PROTECT 的旧工具输出。"""
    total = 0
    pruned = 0
    to_prune = []
    turns = 0

    for msg in reversed(messages):
        if msg.role == "user":
            turns += 1
        if turns < 2:  # 最近 2 轮对话不碰
            continue
        if msg.role == "assistant" and getattr(msg, "summary", False):
            break  # 已有摘要，停止
        for part in msg.parts:
            if part.type == "tool" and part.tool not in PRUNE_PROTECTED_TOOLS:
                estimate = count_tokens(part.output)
                total += estimate
                if total > PRUNE_PROTECT:
                    pruned += estimate
                    to_prune.append(part)

    if pruned > PRUNE_MINIMUM:
        for part in to_prune:
            part.output = "[Old tool result content cleared]"
            part.compacted_at = time.time()
    return pruned
```

**效果**：无 API 调用，纯本地操作。清理"你 10 轮前读的文件内容"不回影响当前任务。

#### 第二层：Compaction — LLM 摘要 + 指针

核心流程参考 `compaction.ts:221-457`：

```
1. select() 决定 head/tail 分界点
   - 计算 tail 预算 = usable_context * 0.25（默认保留 25% 给尾部）
   - 从最后一条 user 消息向前累加 token，直到超出预算
   - 记录 tail_start_id = 保留段第一条 user 消息的 ID

2. 对 head 部分调用 LLM 生成摘要
   - 使用 structured template：
     ## Goal — 用户在做什么
     ## Instructions — 用户的重要指示
     ## Discoveries — 重要发现
     ## Accomplished — 已完成/进行中/待完成
     ## Relevant files/directories — 相关文件列表

3. 摘要存储为 assistant 消息，带 summary: true 标记
   - 这条消息在后续轮次可被 prune() 识别为边界

4. 可选：auto-continue
   - 压缩完成后自动注入 "Continue if you have next steps..."
```

#### 第三层：Emergency Truncation — 连摘要模型都放不下时

```python
def emergency_truncate(messages, model):
    """从最早的消息开始删除，直到 token 数降到 usable_context 以下。"""
    while estimate_tokens(messages) > model.usable_context:
        if not messages:
            raise ContextOverflowError("Even empty context exceeds model limit")
        messages.pop(0)  # 删除最早的消息
    return messages
```

### 3.3 保留预算（preserveRecentBudget）

```python
def preserve_recent_budget(usable_context: int) -> int:
    """保留给最近对话的 token 预算。"""
    MIN_PRESERVE = 2_000
    MAX_PRESERVE = 8_000
    default = int(usable_context * 0.25)
    return max(MIN_PRESERVE, min(MAX_PRESERVE, default))
```

对于 128K 窗口（usable ~108K），tail 保留约 8K token ≈ 最近 2-3 轮对话。

## 四、改进三：KV Cache 优化（参考 CCB）

### 4.1 KV Cache 原理

Anthropic API（以及大多数 LLM API）的 prompt caching 机制：

```
请求 N:
  [system_prompt] [tool_defs] [msg1] [msg2] [msg3(user)]

请求 N+1:
  [system_prompt] [tool_defs] [msg1] [msg2] [msg3(user)] [msg3(asst)] [msg4(user)]
  ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^
  这部分和请求 N 的前缀完全相同 → Cache 命中！
```

**Cache 命中的条件（所有条件必须同时满足）**：
1. System prompt 不变（逐字符）
2. Tool 定义不变（名称、描述、input_schema 都不变）
3. 消息前缀不变（从第一条消息到新消息的起点）
4. Model 不变
5. Thinking/effort 配置不变
6. Beta headers 不变

**任何一个变了 → 整个 Cache 作废，重新计算。**

### 4.2 CCB 的缓存断点检测（728 行）

`promptCacheBreakDetection.ts` 的核心设计：

#### Phase 1（API 调用前）：记录状态快照

```typescript
recordPromptState({
  system: TextBlockParam[],     // 系统提示词
  toolSchemas: BetaToolUnion[], // 工具定义
  querySource: QuerySource,     // 调用来源（主线程/子Agent/压缩）
  model: string,                // 模型名
  fastMode: boolean,            // 快速模式
  globalCacheStrategy: string,  // 缓存策略
  betas: string[],              // Beta headers
  effortValue: string,          // thinking effort
  extraBodyParams: unknown,     // 额外请求体参数
})
```

内部计算所有字段的 hash（djb2/Bun.hash），与上一次对比，检测变化：

```typescript
// 记录所有变化（但不触发告警，等 Phase 2 确认）
prev.pendingChanges = {
  systemPromptChanged: boolean,
  toolSchemasChanged: boolean,
  modelChanged: boolean,
  addedTools: string[],        // 哪些工具新增
  removedTools: string[],      // 哪些工具移除
  changedToolSchemas: string[], // 哪些工具的 schema 变了
  systemCharDelta: number,     // system prompt 字符变化
  // ...
}
```

#### Phase 2（API 调用后）：对比 cache_read_tokens

```typescript
checkResponseForCacheBreak(querySource, cacheReadTokens, cacheCreationTokens, messages)

// 判定逻辑
const tokenDrop = prevCacheRead - cacheReadTokens
if (cacheReadTokens >= prevCacheRead * 0.95 || tokenDrop < 2000) {
  // 正常波动，不算断点
  return
}
// 否则是 cache break，用 pendingChanges 解释原因
```

**关键阈值**：
- `MIN_CACHE_MISS_TOKENS = 2000`：少于 2000 token 的下降不算断点
- 5% 容差：`cacheReadTokens >= prevCacheRead * 0.95` 不算断点
- TTL 检测：超过 5 分钟/1 小时无调用 → 可能是 TTL 过期而非代码问题

#### 特殊处理

```typescript
// 压缩后重置基线（因为消息数减少，cache read 自然会降）
notifyCompaction(querySource, agentId)

// 缓存删除操作也会导致下降（不是断点）
notifyCacheDeletion(querySource, agentId)
```

### 4.3 my-agent 落地设计

my-agent 不需要 728 行的完整实现。核心思路是**让 system prompt 和 tool defs 保持 hash 稳定**：

#### 4.3.1 System Prompt 稳定化

**当前问题**：`cli.py` 中 system_prompt 写死，但 `agent/loop.py` 中 `prompt_builder.build()` 可能每轮动态变化。

**改进**：

```python
# agent/prompt.py — 新增
class PromptBuilder:
    def __init__(self):
        self._last_hash: int | None = None
        self._last_built: str = ""

    def build(self, context: Context) -> str:
        """构建 system prompt。仅当 fragment 变化时重新拼接。"""
        current_hash = self._compute_fragment_hash()
        if current_hash == self._last_hash:
            return self._last_built  # 缓存命中

        parts = []
        for fragment in sorted(self._fragments, key=lambda f: f.priority):
            parts.append(fragment.content)
        self._last_built = "\n\n".join(parts)
        self._last_hash = current_hash
        return self._last_built
```

#### 4.3.2 Tool 定义稳定化

**当前问题**：`tools.get_definitions()` 每轮都重新序列化。

**改进**：ToolRegistry 加缓存：

```python
class ToolRegistry:
    def __init__(self):
        self._defs_cache: list[ToolDefinition] | None = None
        self._defs_dirty = True

    def register(self, tool):
        self._tools[tool.definition.name] = tool
        self._defs_dirty = True  # 标记缓存失效

    def get_definitions(self) -> list[ToolDefinition]:
        if self._defs_dirty or self._defs_cache is None:
            self._defs_cache = [t.definition for t in self._tools.values()]
            self._defs_cache.sort(key=lambda d: d.name)  # 排序保证稳定
            self._defs_dirty = False
        return self._defs_cache
```

#### 4.3.3 cache_control 标记（Anthropic API 专有）

对 Anthropic Provider，在 system prompt 的最后一个 text block 上加 `cache_control`：

```python
# AnthropicProvider._build_system_blocks()
system_blocks = []
# ... 前面的 block ...
system_blocks.append({
    "type": "text",
    "text": dynamic_part,  # 可能变化的动态内容
})
# 最后一个 block 加 cache_control
system_blocks[-1]["cache_control"] = {"type": "ephemeral"}
```

**策略**：
- System prompt 分两段——静态段（cacheable）+ 动态段（不加 cache_control）
- 静态段包含：Agent 身份、能力描述、工具列表
- 动态段包含：当前时间、git status、CLAUDE.md 内容（这些会变）

#### 4.3.4 简化的缓存断点检测（可选）

```python
class CacheBreakDetector:
    """简化的单阶段检测——在 API 响应后对比。"""

    def __init__(self):
        self._prev_system_hash: int | None = None
        self._prev_tools_hash: int | None = None
        self._prev_cache_read: int | None = None

    def record(self, system_prompt: str, tools: list[ToolDefinition]):
        self._prev_system_hash = hash(system_prompt)
        self._prev_tools_hash = hash(json.dumps([t.name for t in tools], sort_keys=True))

    def check(self, cache_read_tokens: int) -> str | None:
        if self._prev_cache_read is None:
            self._prev_cache_read = cache_read_tokens
            return None

        drop = self._prev_cache_read - cache_read_tokens
        self._prev_cache_read = cache_read_tokens

        if drop > 2000 and cache_read_tokens < self._prev_cache_read * 0.95:
            return f"Cache break detected: {drop} tokens lost"
        return None
```

## 五、my-agent 完整上下文管理架构

### 5.1 整体流程

```
AgentLoop.run()
  │
  ├─ 每轮开始
  │   ├─ 1. estimate_tokens() → 估算当前上下文 token 数
  │   ├─ 2. 如果超出 usable_context：
  │   │   ├─ 第一层：prune_old_tool_results()      ← 无 API 调用
  │   │   │   如果释放 >= PRUNE_MINIMUM → 继续
  │   │   ├─ 第二层：compact_with_summary()         ← 1 次 API 调用
  │   │   │   生成摘要 + tail_start_id 指针
  │   │   └─ 第三层：emergency_truncate()           ← 无 API 调用，保底
  │   ├─ 3. build_system_prompt() → 使用 hash 缓存
  │   └─ 4. get_tool_definitions() → 使用排序缓存
  │
  ├─ LLM 调用
  │   └─ 5. 记录 cache_read_tokens
  │       └─ 如果显著下降 → log warning（不阻塞）
  │
  └─ 工具执行
      └─ 限制单个工具结果 max_tool_result_chars（已有）
```

### 5.2 新增文件清单

| 文件 | 职责 | 行数估算 |
|------|------|---------|
| `agent/context_window.py` | Token 估算、溢出检测、usable_context 计算 | ~80 |
| `agent/compaction.py` | Prune + compact + emergency truncate 三层 | ~200 |
| `agent/cache_break.py` | 简化的缓存断点检测（可选） | ~60 |

### 5.3 修改文件清单

| 文件 | 改动 |
|------|------|
| `ai/provider.py` | `IModelProvider` 加 `context_window`、`usable_context` 属性；各实现类提供具体值 |
