## 二十、多 Agent 协作系统

> **核心思想**：Agent 不是孤军奋战。当任务复杂到需要并行处理、专项分工时，
> Agent 可以派生子 Agent（线程级隔离），子 Agent 完成后结果汇总回父 Agent。
>
> **V1 策略**：Hermes 线程级委托（最轻量、最快启动），预留进程级隔离插口。

### 设计目标

1. **基础委托**：父 Agent 可以派生一个子 Agent 执行特定任务
2. **并行执行**：多个子 Agent 同时运行，结果汇总
3. **并发控制**：最大并发数 + 深度限制，防止无限递归
4. **上下文隔离**：每个子 Agent 拥有独立的上下文和工具集
5. **结果聚合**：子 Agent 完成后，结果和 token 用量汇总回父 Agent

### 参考来源与设计取舍

| 参考项目 | 隔离方式 | 我们的选择 |
|----------|---------|-----------|
| Hermes Agent | 线程池（ThreadPoolExecutor） | **V1 采用** — 启动最快，Python 原生 asyncio |
| multica | Go goroutine + 信号量 | 参考并发控制模式（信号量 + WaitGroup） |
| Pi Agent | OS 进程 + JSON 流式 | V2 预留 — 更强隔离但启动慢 |
| CCB | Fork + Worktree | V3 预留 — 最复杂但功能最全 |
| oh-my-opencode | 进程内 delegate_task | 参考角色定义和选择逻辑 |

### 代理类型定义

```python
@dataclass
class AgentTypeConfig:
    """代理类型配置。定义一种 Agent 角色。

    参考：
    - oh-my-opencode 的 agent 定义（Sisyphus/Atlas/Oracle 等）
    - CCB 的 Agent YAML frontmatter（name/model/tools/systemPrompt）
    - Hermes 的子 Agent 构建（provider/model/toolsets/system_prompt）
    - multica 的 Runtime/Provider 映射

    存储位置：.agent-base/agents/{name}.yaml
    """
    name: str                          # Agent 类型名（如 "coder", "reviewer", "explorer"）
    description: str                   # 一句话描述（用于 Agent 选择匹配）
    model: str = ""                    # 指定模型（空 = 使用父 Agent 的模型）
    system_prompt: str = ""            # 子 Agent 的系统提示词（聚焦版）
    tools_whitelist: list[str] = field(default_factory=list)
                                       # 工具白名单（空 = 继承父 Agent 的全部工具）
    tools_blacklist: list[str] = field(default_factory=lambda: [
        "delegate_task",               # 子 Agent 不能再委托（防止无限递归，参考 Hermes）
        "crystallize",                 # 子 Agent 不能触发结晶（只有主 Agent 可以）
    ])
    # 注意：web_fetch / web_search / web_browser_* 默认不在黑名单中
    # 子 Agent 可以联网和操作浏览器。如需限制，手动添加到 tools_blacklist。
    max_tokens: int = 4096            # 子 Agent 单次最大输出 token
    category: str = "general"         # 分类（用于 Agent 选择匹配）
    when_to_use: str = ""             # 触发条件示例（参考 CCB agent description + Superpowers example）
                                       # 用于 Agent 选择匹配，如："当需要审查已完成的项目步骤时"
    initial_prompt: str = ""          # 首轮前置指令（参考 CCB initialPrompt）
                                       # 注入到子 Agent 的第一个 user turn，如 slash command 指令
    omit_context: bool = False        # 是否省略完整 CLAUDE.md 上下文（参考 CCB omitClaudeMd）
                                       # 只读 Agent（如 explorer/reviewer）设为 True，
                                       # 节省 ~5-15K token/次（参考 CCB 数据：34M+ Explore spawns）

    # —— 发现机制说明 ——
    # Agent 类型按优先级发现（参考 CCB loadAgentsDir.ts 6层覆盖机制）：
    #   builtIn → plugin → userSettings → projectSettings → flagSettings → managedAgents
    # 后发现的同名 agentType 覆盖先发现的。
    # V1 简化：register_agent_type() 注册 + .agent-base/agents/ 目录扫描。
    # V2 预留：分层发现 + requiredMcpServers 前置检查（参考 CCB filterAgentsByMcpRequirements）

    # —— 预留插口（注释标注，后续通过插件实现）——
    # V2 权限模式（参考 CCB permissionMode）：
    #   permission_mode: Literal["bubble", "default", "restricted"] = "restricted"
    #   bubble = 子 Agent 权限请求浮到父终端
    #   restricted = 子 Agent 使用受限工具集，不请求权限
    #
    # V2 隔离级别（参考 CCB isolation）：
    #   isolation: Literal["thread", "process", "worktree"] = "thread"
    #   thread = asyncio 并发（V1 默认，最快）
    #   process = 子进程隔离（V2，更强隔离）
    #   worktree = git worktree 隔离（V3，文件系统级隔离）
    #
    # V2 MCP 配置（参考 CCB agent frontmatter）：
    #   mcp_servers: list[str] = field(default_factory=list)
    #   子 Agent 使用的 MCP 服务器列表
```

### 接口定义

```python
@dataclass
class SubAgentResult:
    """子 Agent 执行结果。"""
    task_id: str                       # 任务 ID
    agent_type: str                    # 使用的 Agent 类型
    status: str                        # "completed" / "failed" / "timeout"
    output: str                        # 子 Agent 的最终输出
    usage: dict                        # token 用量 {input_tokens, output_tokens, cost}
    duration_ms: int                   # 执行时长
    error: str | None = None          # 错误信息（如果失败）
    exit_reason: str = ""              # 退出原因:"completed" / "max_iterations" / "timeout"
    tool_trace: list = field(default_factory=list)
                                        # 工具调用轨迹（参考 Hermes delegate_tool.py 535-569行）
                                        # [{tool, args_bytes, result_bytes, status}, ...]
                                        # 供父Agent分析子Agent做了什么
class IAgentRuntime(ABC):
    """多 Agent 运行时接口。

    职责边界：
    - 代理类型注册：管理可用的 Agent 类型配置
    - 子 Agent 创建：根据类型创建隔离的子 Agent 实例
    - 并发控制：限制最大并发数和递归深度
    - 结果聚合：收集子 Agent 的输出和 token 用量
    - 不负责"何时委托"（那是 Agent 自己的决策）
    - 不负责"工具执行"（那是 ITool 的职责）

    参考：
    - Hermes 的 delegate_tool.py（线程池 + 凭证池 + 受限工具集）
    - multica 的 daemon.go（pollLoop 信号量 + handleTask）
    - Pi Agent 的 subagent 扩展（Single/Parallel/Chain 三种模式）
    - oh-my-opencode 的 delegate_task 工具（Agent 选择 + 上下文传递）

    V1 实现：asyncio 线程级隔离（Hermes 模式）
    V2 预留：OS 进程级隔离（Pi 模式）— 通过 isolation 字段切换
    """

    # —— 代理类型注册 ——

    @abstractmethod
    def register_agent_type(self, config: AgentTypeConfig) -> None:
        """注册一个代理类型。

        参考 oh-my-opencode 的 createBuiltinAgents()。
        参考 CCB 的 .claude/agents/ 目录扫描。
        代理类型配置持久化到 .agent-base/agents/{name}.yaml。

        验证：
        - name 不能与已注册的重复
        - tools_whitelist 中的工具必须存在于 ICapabilityCatalog
        """
        ...

    @abstractmethod
    def get_agent_type(self, name: str) -> AgentTypeConfig | None:
        """获取指定代理类型的配置。"""
        ...

    @abstractmethod
    def list_agent_types(self) -> list[AgentTypeConfig]:
        """列出所有已注册的代理类型。"""
        ...

    # —— 单任务委托 ——

    @abstractmethod
    async def spawn(self, agent_type: str | None = None, task: str = "",
                    context: dict | None = None) -> SubAgentResult:
        """派生一个子 Agent 执行任务（同步等待结果）。

        agent_type 可选：None 时自动调用 select_agent(task) 选择最匹配类型。

        前置检查（参考 Hermes delegate_tool.py 709-716行）：
        0. agent_type 为空时 → select_agent(task) 自动选择
        0. 检查 depth >= max_depth → 直接返回 SubAgentResult(status="rejected")

        运行时安全（参考 CCB forkSubagent.ts isInForkChild() 78-89行）：
        - 不仅入口检查 depth，执行过程中也验证子 Agent 是否尝试越权
        - V1: 在 spawn() 入口处检查 depth（参考 Hermes）
        - V2: 运行时检测子 Agent 消息中的递归委托标记（参考 CCB 的 boilerplate tag 扫描）

        子 Agent 行为约束（参考 CCB forkSubagent.ts buildChildMessage() 171-198行）：
        - 通过 initial_prompt 注入结构化行为规则（非协商）
        - 包含：执行范围、输出格式、禁止事项
        - 输出格式：Scope / Result / Key files / Files changed / Issues

        流程（参考 Hermes delegate_tool + multica handleTask）:
        1. 检查并发限制（active_children < max_concurrent_children）
        2. 检查深度限制（current_depth < max_depth，参考 Hermes MAX_DEPTH=2）
        3. 从 AgentTypeConfig 构建子 Agent 上下文：
           - system_prompt = config.system_prompt（聚焦版提示词）
           - tools = 按 whitelist/blacklist 过滤父 Agent 的工具集
           - model = config.model 或继承父 Agent 的模型
        4. 启动子 Agent 循环（复用 agent/ 层的循环逻辑）
        5. 触发 IHook.on_event("subagent_start")
        6. 等待子 Agent 完成
        7. 触发 IHook.on_event("subagent_stop")
        8. 收集结果 + token 用量
        9. 返回 SubAgentResult

        context 可包含：
        - "parent_messages": 父 Agent 最近 N 条消息（供子 Agent 理解上下文）
        - "project_path": 项目路径
        - "relevant_memory": 从 IMemoryBackend 预取的相关记忆

        安全约束（参考 Hermes 子 Agent 禁止工具）：
        - 子 Agent 默认不能使用 delegate_task（防止递归委托）
        - 子 Agent 默认不能使用 crystallize（只有主 Agent 可以触发结晶）
        - 子 Agent 不能操作主 Agent 的记忆（独立的记忆命名空间）

        并发安全：
        - 内部由 asyncio.Lock 保护（active_children 计数、任务状态）
        """
        ...

    # —— 并行委托 ——

    @abstractmethod
    async def spawn_parallel(
        self,
        tasks: list[dict],
        max_concurrency: int = 3
    ) -> list[SubAgentResult]:
        """并行派生多个子 Agent（参考 Pi Parallel + multica pollLoop）。

        tasks 格式：[{"agent_type": str, "task": str, "context": dict}, ...]

        并发控制（参考 multica 的有界信号量）：
        - 使用 asyncio.Semaphore(max_concurrency) 控制并发数
        - 默认 max_concurrency=3（参考 Hermes delegation.max_concurrent_children）
        - 所有任务完成后汇总结果

        参考 Pi Agent 的 mapWithConcurrencyLimit(tasks, 4)。
        """
        ...

    # —— Agent 自动选择 ——

    @abstractmethod
    def select_agent(self, task: str) -> str | None:
        """根据任务描述自动选择最匹配的 Agent 类型。

        匹配算法：Overlap Coefficient —— |A ∩ B| / min(|A|, |B|)
        对中文友好：通过 _tokenize() 做中文分词后再计算。

        流程：
        1. 对 task 分词 → task_tokens
        2. 遍历所有注册的 AgentTypeConfig，对 (name + description + when_to_use + category) 分词
        3. 计算 overlap 得分，返回最高且 >= 0.3 的 agent name
        4. 所有 < 0.3 → 返回 None（由 LLM 自行决定或使用默认类型）

        V2 预留：用 embedding 替代词重叠匹配。
        """
        ...

    # —— 状态查询 ——

    @abstractmethod
    def get_active_children(self) -> list[dict]:
        """返回当前活跃的子 Agent 列表。

        [{task_id, agent_type, status, started_at}]
        用于 IFlowInspector 监控和前端可视化。
        """
        ...

    @abstractmethod
    def get_delegation_stats(self) -> dict:
        """返回委托统计信息。

        {total_spawned, total_completed, total_failed,
         total_input_tokens, total_output_tokens, total_cost,
         active_count, max_depth_reached}

        用于 IPromptBuilder 的 AGENT_USAGE 片段和 IMetaAgent 的性能评估。
        """
        ...

    # —— 配置 ——

    @abstractmethod
    def get_config(self) -> dict:
        """返回运行时配置。

        {
            max_concurrent_children: 3,     # 最大并发子 Agent
            max_depth: 2,                   # 最大递归深度
            default_timeout_sec: 300,       # 子 Agent 默认超时
            heartbeat_interval_sec: 30,     # 心跳间隔（参考 Hermes 心跳线程）
        }

        并发安全规则（参考 Hermes delegate_tool.py skip_memory=True + DELEGATE_BLOCKED_TOOLS）：
        1. Nudge 只在主Agent的 turn_end 触发
        2. 子Agent的 tools_blacklist 默认包含 memory 工具
        3. 子Agent不触发、不接收、不响应 Nudge

        预留插口：
        
        配置更新策略（参考 Hermes delegate_tool.py _load_config() 1086-1106行）：
        V1 采用懒加载：每次 spawn() 调用时重新读取 AgentTypeConfig。
        配置文件变更后，下次 spawn 自动生效，无需热重载接口。
        V2 预留：IHotLoader 文件监听 + 热重载回调。
        # V2 凭证池（参考 Hermes CredentialPool）：
        #   credential_pool_size: int = 5
        #   credential_pool: ICredentialPool | None = None
        # V2 进程级隔离配置：
        #   default_isolation: str = "thread"
        #   process_timeout_sec: int = 600
        """
        ...

    # —— 预留插口（注释标注，后续通过插件实现）——
    #
    # V2 链式委托（参考 Pi Agent Chain 模式）：
    #   async def spawn_chain(self, chain: list[dict]) -> list[SubAgentResult]:
    #       """链式委托：步骤间传递上下文。

## 二十一、自进化与沉淀系统

> **核心思想**：Agent 不仅执行任务，还从成功经验中学习。
> 完成"部署 Django 到 AWS"后，下次遇到类似任务可以直接复用经验。
>
> **V1 策略**：GenericAgent 的自主结晶理念 + Hermes 的 Nudge 提醒机制。
> 基础能力：工作检查点 + 结晶触发 + 质量评分 + Nudge。

### 设计目标

1. **工作检查点**：会话内的关键信息跨轮次保持（不丢失上下文）
2. **结晶触发**：任务成功完成后评估是否值得固化为技能
3. **质量评分**：结晶前评估技能的清晰度/完整度/可操作性
4. **Nudge 机制**：定期提醒 Agent 使用记忆和技能工具
5. **长期更新**：跨会话的记忆沉淀和技能积累

## 二十八、Agent 间通信机制

> **核心思想**：子Agent 在执行过程中可能需要与其他 Agent 通信——向父Agent 汇报中间发现、
> 向兄弟 Agent 请求信息、或父Agent 向子Agent 注入新指令。
>
> **V1 策略**：asyncio.Queue 收件箱模式——每个 Agent 有一个消息队列，通过 `agent_message` 工具投递。

### 设计目标

1. **子Agent → 父Agent**：回传中间发现、请求更多上下文
2. **父Agent → 子Agent**：spawn 后注入额外指令或修正方向
3. **子Agent ↔ 子Agent**：并行任务间协调（V1 基础支持，V2 增强）

### 实现方式

```python
# IAgentRuntime 中的消息基础设施

class AgentRuntime:
    _inboxes: dict[str, asyncio.Queue]   # ★ 每个 Agent 一个收件箱

    def send_message(self, from_agent: str, to_agent: str, content: str) -> None:
        """向指定 Agent 发送消息。非阻塞投递。"""
        ...

    def check_inbox(self, agent_id: str) -> str | None:
        """检查收件箱，返回最早未读消息。非阻塞。"""
        ...
```

### 消息流

```
子Agent 调用 agent_message 工具
  → AgentMessageTool.execute()
    → AgentRuntime.send_message(from=子Agent, to="main", content=发现)
      → _inboxes["main"].put_nowait(msg)
        → 父Agent 的 AgentLoop 在下轮内层循环开始时
          → AgentRuntime.check_inbox("main")  → 返回消息
            → 作为 [子Agent消息] 注入 context.messages
```

### 工具定义

| 工具 | 名称 | 用途 |
|------|------|------|
| `AgentMessageTool` | `agent_message` | 向指定 Agent 发送消息。参数：to_agent（目标ID）、content（内容）、message_type（info/request/result） |

### 设计约束

- 消息通过 asyncio.Queue 传递，不持久化（会话内通信，会话结束自动清理）
- 每个 inbox Queue 默认 maxsize=50（防止内存泄漏）
- AgentMessageTool 不在子Agent 黑名单中（允许子Agent 间通信）
- 父Agent 的 AgentLoop 在每轮内层循环开始时检查 inbox（非阻塞）

## 二十九、中文分词搜索增强

> **核心思想**：V1 中文搜索不能依赖简单的空格分词——中文没有天然空格分隔。
> 引入 jieba 分词提升搜索召回率，同时保留零依赖回退路径。
>
> **影响范围**：IMemoryBackend 的 `search_keywords()` 和 IAgentRuntime 的 `select_agent()`。

### 设计目标

1. **中文分词**：用 jieba 精确模式对中文查询做分词
2. **回退机制**：jieba 不可用时自动回退到字符二元组（bigram），零依赖风险
3. **统一入口**：`_tokenize()` 函数供 MemoryIndex 和 AgentRuntime 共用

### 实现方式

```python
@staticmethod
def _tokenize(text: str) -> list[str]:
    """中文分词 + 英文分词统一入口。"""
    try:
        import jieba
        # jieba 精确模式分词
        words = jieba.lcut(text.lower())
        return [w.strip() for w in words if w.strip() and not re.match(r'^[\s\W_]+$', w)]
    except ImportError:
        pass
    # 回退：字符二元组（bigram）
    # 英文部分：按正则提取 [a-z0-9]+
    # 中文部分：逐字符二元组，单字直接保留
    ...
```

### 搜索改进

| 维度 | 改进前（空格分词） | 改进后（jieba） |
|------|-------------------|-----------------|
| 中文查询 "部署 Django 到 AWS" | 拆分为 "部署", "Django", "到", "AWS" → "到" 是无意义停用词 | jieba 精准切分 → "部署", "Django", "AWS" |
| 回退路径 | 无（jieba 不可用则搜索失败） | 自动回退到 bigram → "部署"→"Django"→"AWS"（召回率下降约 30%） |
| 英文混合查询 | 按空格+标点拆分 | 正则提取 [a-z0-9]+ 单词 + 中文部分 jieba |
| 结果排序 | 简单去重，无排序 | 按命中次数降序排序 |

### 性能考量

- jieba 首次加载词典约 100ms（一次性成本）
- 后续每次分词 < 1ms（对搜索延迟几乎无影响）
- 建议生产环境 `pip install jieba` 确保最佳召回率

## 三十、Agent 自动选择

> **核心思想**：当 Agent 需要委托子任务时，不需要手动指定 agent_type——
> IAgentRuntime 根据任务描述自动匹配最合适的 Agent 类型。
>
> **V1 策略**：Overlap Coefficient 词重叠匹配——快速、免费、可解释。

### 设计目标

1. **自动匹配**：`spawn(task="重构 auth 模块")` 不加 agent_type 时自动选择
2. **中文友好**：通过 `_tokenize()` 做中文分词后再计算匹配度
3. **回退保护**：所有类型匹配度 < 0.3 时返回 None，由 LLM 自行决定

### 匹配算法

```
Overlap Coefficient = |A ∩ B| / min(|A|, |B|)

A = task 的分词集合
B = (agent_name + description + when_to_use + category) 的分词集合

与 Jaccard 的区别：Overlap Coefficient 对长度差异容忍度更好——
短 task 如"审查代码"与长 description 也匹配良好。
```

### 接口签名

```python
def select_agent(self, task: str) -> str | None:
    """根据任务描述自动选择最匹配的 Agent 类型。
    
    返回：匹配度最高且 >= 0.3 的 agent name
    返回 None：无匹配（由 LLM 自行决定或使用默认类型）
    """
```

### spawn() 签名变更

```python
# 改进前：agent_type 必传
async def spawn(self, agent_type: str, task: str, ...) -> SubAgentResult:

# 改进后：agent_type 可选，None 时自动选择
async def spawn(self, agent_type: str | None = None, task: str = "", ...) -> SubAgentResult:
```

### 使用示例

```python
# 方式1：显式指定（保持兼容）
result = await runtime.spawn(agent_type="coder", task="写一个用户认证模块")

# 方式2：自动选择（★ 新增）
result = await runtime.spawn(task="写一个用户认证模块")
# → select_agent("写一个用户认证模块") → "coder"（匹配度 0.65）

# 方式3：在 spawn_parallel 中混用
results = await runtime.spawn_parallel([
    {"agent_type": "reviewer", "task": "审查 PR #42"},
    {"task": "搜索数据库迁移相关文件"},  # 自动选择 → "explorer"
])
```

### 验收标准

| # | 验收项 |
|---|--------|
| 1 | `select_agent("重构认证模块")` 匹配到 "coder"（score >= 0.3） |
| 2 | `select_agent("查找所有 TODO 注释")` 匹配到 "explorer" |
| 3 | `select_agent("xyzzy 无意义查询")` 返回 None |
| 4 | `spawn(task="写代码")` 不加 agent_type 时自动选择并成功执行 |

## 三十一、CLAUDE.md 上下文加载机制

> **核心思想**：借鉴 Claude Code 的 CLAUDE.md 层级加载机制，Agent 在启动时自动发现并加载
> 项目中的 CLAUDE.md 文件作为行为指南和项目上下文。用户可以在项目根目录或任意子目录
> 放置 CLAUDE.md 文件，Agent 自动从项目根到当前目录逐级加载。
>
> **参考来源**：
> - **CCB** `claudemd.ts`：层级遍历 + 多路径 + 优先级排序 + @include 指令
> - **andrej-karpathy-skills** `CLAUDE.md`：纯 Markdown 行为指南，无 frontmatter，简洁可读
> - **opencode** `.opencode/rules/`：项目级规则文件自动加载

### 设计目标

1. **零配置自动加载**：Agent 启动时自动发现 CLAUDE.md，无需用户手动指定
2. **层级覆盖**：项目根目录的 CLAUDE.md 提供全局上下文，子目录的可补充或覆盖
3. **用户级 + 项目级**：`~/.agent/CLAUDE.md` 提供跨项目全局指南，项目级文件提供项目特定指南
4. **V1 极简**：不做 @include、不做 frontmatter 解析、不做条件规则

### 发现优先级（从低到高）

```
优先级  来源                    路径                                 说明
──────────────────────────────────────────────────────────────────────────
  0     用户全局指南            ~/.agent/CLAUDE.md                   跨项目行为规范
  1     项目根目录              {project_root}/CLAUDE.md              项目级共用指南
  2     项目根目录(私)          {project_root}/.agent/CLAUDE.md      项目级个人私有
  ...   逐级向上到 CWD          每级目录重复上述检查                   子目录补充
  N     CWD 目录               {cwd}/CLAUDE.md                      当前目录（最高优先级）
```

**关键规则**：
- 后加载的优先级更高——CWD 下的 CLAUDE.md 比根目录的更"近"，能覆盖上级设定
- 同级目录下 `CLAUDE.md` 和 `.agent/CLAUDE.md` 同时存在时，两份都加载
- `CLAUDE.md` 适合提交到 git（团队共享），`.agent/CLAUDE.md` 适合 gitignore（个人私有）
- 找到 `.git` 目录时停止向上遍历（以 git 仓库为项目根）

### 组装格式

多个 CLAUDE.md 文件被组装为一个整体，注入 system prompt（优先级最高的 PromptFragment）：

```
# CLAUDE.md

Contents of ~/.agent/CLAUDE.md (user's global instructions):

<用户全局行为指南>

---

Contents of /project/CLAUDE.md (project instructions):

<项目级共用指南>

---

Contents of /project/src/CLAUDE.md (project instructions):

<子目录补充上下文>
```

### CLAUDE.md 文件示例

参考 andrej-karpathy-skills 的风格，一个典型的项目 CLAUDE.md 长这样：

```markdown
# CLAUDE.md

This file provides guidance to Claude when working with code in this repository.

## 项目概述

这是一个 Python Web 应用，使用 FastAPI + SQLAlchemy + PostgreSQL。

## 开发命令

```bash
pip install -e ".[dev]"
pytest --cov=src tests/
ruff check src/
```

## 架构约定

- 三层架构：routes/ → services/ → repositories/
- 依赖注入用 fastapi Depends()
- 数据库迁移用 Alembic

## 行为准则

- 修改数据库模型后必须同步更新迁移文件
- 新增 API 端点必须添加测试
- 禁止直接操作数据库——必须通过 Repository 层
```

### 安全约束

| 约束 | 值 | 说明 |
|------|-----|------|
| 单文件最大 | 50 KB | 防止 CI 配置等大文件被误加载 |
| 编码 | UTF-8 | 非 UTF-8 文件静默跳过 |
| 遍历上限 | 文件系统根 | 遇到 .git 停止（以仓库为界） |
| 内容缓存 | context.metadata | 首次加载后缓存，避免每轮重复 I/O |

### 集成到 Prompt

CLAUDE.md 内容作为 `CLAUDE_MD` 片段注入 IPromptBuilder：
- `priority = 0`（最高优先级，在所有其他片段之前呈现给模型）
- 属于核心片段（0-24），不可被 token 预算裁剪
- 通过 `IPromptBuilder.build()` 自动组装到 system prompt 最前面

### V2 预留

| 扩展 | 说明 |
|------|------|
| `@include` 指令 | 支持 `@path/to/file` 引用其他文件（参考 CCB） |
| 条件规则 | frontmatter `paths` glob 匹配——只有操作匹配文件时才注入规则 |
| 文件变更监听 | 通过 IHotLoader 监听 CLAUDE.md 变更，自动刷新缓存 |
| `.agent/rules/` 目录 | 递归加载规则目录下的所有 `.md` 文件（参考 CCB rules/） |

### 验收标准

| # | 验收项 |
|---|--------|
| 1 | `discover_claude_mds()` 返回按 priority 排序的文件列表 |
| 2 | 用户级 `~/.agent/CLAUDE.md` 存在时被加载 |
| 3 | 项目根 `CLAUDE.md` 存在时被加载 |
| 4 | 子目录 `CLAUDE.md` 存在时被加载（优先级高于根目录） |
| 5 | 找不到 .git 时以 CWD 为项目根 |
| 6 | 单文件超过 50KB 时被跳过（不报错） |
| 7 | `assemble_claude_md_content()` 输出格式正确（带路径和描述） |
| 8 | CLAUDE_MD 片段注入到 system prompt（priority=0） |
